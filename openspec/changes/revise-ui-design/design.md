## Context

本次变更服务于 workflowRunId `6b74f1b8-1725-4755-877f-ea35e6436020` 的 UI 设计阶段修订。当前项目为“移动端应用：会议室预约系统”，已有阶段产物包括交互式 HTML 原型、桌面/移动端 SVG、组件库、交互说明、响应式适配文档和 UI 设计规范。用户尚未填写具体反馈，因此本次设计采用“受控轻量修订”策略：先用 OpenSpec 明确可改范围、默认修订原则和验收要求，再由 `/opsx:apply` 对允许范围内的 UI 文档与静态原型进行检查和必要修订。

## Goals / Non-Goals

**Goals:**
- 将 UI 设计阶段修订限制在 `docs/移动端应用：会议室预约系统/UI原型/` 与 `docs/移动端应用：会议室预约系统/产品设计/03-UI设计规范/UI-Design-Spec.md`。
- 对现有原型与说明做一致性检查和必要修订，使设计稿、组件库、交互说明、响应式适配与 UI 设计规范中的页面名称、组件名称、状态文案、业务规则和文件引用保持一致。
- 在无具体反馈时，优先改善移动端会议室预约核心路径的可读性、触控可用性、异常/空状态表达和静态文档说明完整性。
- 保留当前 UI 阶段已覆盖的会议室预约业务范围，包括预约、审批、服务确认、扫码签到、自动释放、取消、转让、二维码维护、空态/异常态和响应式适配。

**Non-Goals:**
- 不新增产品需求、角色、后端接口、数据库字段或运行时工作流。
- 不修改 `apps/`、`packages/`、`tests/` 或非 UI 设计阶段文档。
- 不重做整体视觉风格、信息架构或业务流程。
- 不改变会议室预约系统既有业务规则，例如 30 天可预约窗口、15 分钟时间粒度、超 2 小时审批、开始前 15 分钟签到/取消/转让限制等。

## Decisions

1. **以文档和静态原型为唯一修订对象。**
   - 选择：仅在允许范围内更新 HTML/SVG/Markdown UI 产物与 UI 设计规范。
   - 理由：当前操作是阶段产物修订，不应触发实现层变更。
   - 替代方案：同步修改前端 Vue 组件。拒绝原因：超出允许范围，并可能引入未验证实现风险。

2. **采用一致性优先的默认修订策略。**
   - 选择：优先核对 UI 设计规范、组件清单、交互说明、响应式文档和设计稿中的页面名称、状态文案、业务规则、路径引用。
   - 理由：用户未给出具体反馈时，一致性修订最符合阶段修订诉求，且不会引入新范围。
   - 替代方案：大幅重做视觉表现。拒绝原因：缺少用户反馈支撑，且容易造成不必要的大范围改动。

3. **移动端体验优先，桌面稿作为辅助展示。**
   - 选择：重点检查移动端 H5 的底部导航、安全区、固定操作区、44px 触控热区、状态可读性和错误提示；桌面 SVG 保持为全局结构展示。
   - 理由：项目定位为移动端应用，桌面稿不是核心交付场景。
   - 替代方案：桌面端与移动端同等深度修订。拒绝原因：不符合移动端优先场景。

4. **不因无反馈而新增业务能力。**
   - 选择：仅补强已有状态、文案、引用和交互说明，不新增“日历同步”“会议邀请”等新能力。
   - 理由：避免 UI 修订阶段反向改变产品范围。

## Risks / Trade-offs

- [用户反馈为空导致修订方向不够具体] → 以一致性、可访问性、移动端可用性和文档引用准确性作为默认修订标准。
- [静态 SVG 与 HTML 存在重复维护成本] → 应用阶段需逐项同步文案和状态，避免同一状态在不同产物中含义不一致。
- [过度视觉重构影响已有交付稳定性] → 限制为小而明确的修订，不重做整体信息架构和视觉系统。
- [误改非允许范围文件] → 任务中显式列出可修改路径，并在完成前检查变更文件列表。

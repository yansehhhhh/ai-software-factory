## Context

本次修订发生在“移动端应用：会议室预约系统”的完成前验证阶段。现有生成前端在应用启动 Store 中并行加载普通用户数据和管理员看板数据，导致普通用户也会请求 `/api/hx-meeting/v1/admin/dashboard` 并得到 403；预约查询页面在 `onMounted` 时自动调用 `searchRooms`，导致用户只是进入预约菜单也会默认请求 `/api/hx-meeting/v1/rooms`。

约束条件：修改范围仅限 `docs/移动端应用：会议室预约系统/测试/` 与 `generated/HX-Meeting/frontend`、`generated/HX-Meeting/backend`。本次反馈属于前端请求时机与验证产物修订，不需要变更后端接口契约。

## Goals / Non-Goals

**Goals:**
- 普通用户进入系统或预约相关页面时，不触发管理员看板接口请求。
- 管理员看板数据只在确认当前用户包含 `ADMIN` 角色后加载。
- 进入预约菜单时不自动查询会议室；查询会议室必须由用户点击“查询可用会议室”触发。
- 完成前验证产物补充网络请求回归断言，明确上述行为可验证。

**Non-Goals:**
- 不调整 `/api/hx-meeting/v1/admin/dashboard` 的后端鉴权语义，普通用户访问仍可返回 403。
- 不删除预约页面的手动会议室查询能力。
- 不重构整体路由、权限体系或 UI 视觉样式。
- 不新增新的测试框架或运行时依赖。

## Decisions

1. **启动加载拆分为“先用户、后按角色加载”**
   - 决策：`bootstrap` 先调用 `authApi.me()` 获取当前用户，再并行加载普通用户必需数据；只有 `roles` 包含 `ADMIN` 时才调用 `adminApi.dashboard()`。
   - 理由：管理员接口是否可调用依赖用户角色，必须先知道身份再决定请求集合。
   - 替代方案：保留并行请求并忽略 403。该方案仍违反“非管理员角色不要请求”的反馈，因此不采用。

2. **预约页改为显式查询**
   - 决策：移除 `BookingSearchView.vue` 的 `onMounted(searchRooms)`，保留筛选表单和按钮触发的 `searchRooms`。
   - 理由：页面访问和查询是两个不同用户意图，只有点击查询才应触发 `/rooms`。
   - 替代方案：进入页面延迟自动查询或加 debounce。该方案仍属于默认请求，不符合反馈，因此不采用。

3. **验证用例以网络请求捕获为准**
   - 决策：在 E2E 中监听页面请求，断言普通用户启动过程未出现 `/admin/dashboard`，并断言进入 `/booking` 后点击查询前未出现 GET `/rooms`。
   - 理由：用户反馈聚焦“不要请求”，最直接的验证维度是浏览器实际网络请求，而不是仅检查页面文案。
   - 替代方案：只更新文档说明。该方案缺少可执行回归保护，因此不采用。

## Risks / Trade-offs

- [Risk] 管理员首页依赖 `adminDashboard` 初始化数据，拆分加载后若未在管理员路径补充加载可能出现空态。 → Mitigation：保留 Store 中管理员角色加载逻辑，并在管理员相关页面使用已有 Store 数据或必要时显式加载。
- [Risk] 预约页不自动查询后，空态文案可能让用户误以为无结果。 → Mitigation：测试产物应要求点击查询前展示引导/空态，点击后正常加载列表。
- [Risk] E2E 网络监听如果统计历史请求可能产生误判。 → Mitigation：在进入目标页面前初始化请求数组，并按 URL 与 method 精确过滤。

## Migration Plan

1. 修改生成前端 Store 的启动请求编排，确保管理员接口按角色门控。
2. 修改预约查询页面，取消 mounted 自动查询，仅保留按钮触发。
3. 修订 E2E 和测试文档，增加两个回归点：普通用户不请求管理员看板、进入预约页不默认请求会议室列表。
4. 运行完成前验证命令，更新测试报告中的执行结果。
5. 若发现问题，回滚前端对应修改即可恢复原行为；后端无需迁移。

## Context

当前修订属于“移动端应用：会议室预约系统”的完成前验证阶段缺陷闭环。现有联调报告显示前端 `generated/HX-Meeting/frontend` 已通过真实后端 `generated/HX-Meeting/backend` 完成基础预约、通知列表、管理员看板与会议室列表验证，但用户反馈暴露出三个未覆盖的交互缺陷：

1. 会议室管理页“新增”点击后仍停留在样式/演示层，无法完成真实新增会议室。
2. 底部导航切换存在不稳定渲染，页面偶发只剩标题、正文空白。
3. 通知列表进入详情页发生白屏报错。

约束：本次只能修改 `docs/移动端应用：会议室预约系统/测试/`、`generated/HX-Meeting/frontend` 与 `generated/HX-Meeting/backend`。完成前验证必须以真实 HTTP 请求和页面交互作为依据，不能用 Playwright mock 或前端 fixture 替代接口验收。

## Goals / Non-Goals

**Goals:**

- 会议室管理新增具备真实可用的端到端路径：表单校验、提交真实后端接口、成功后列表刷新并展示新增会议室。
- 底部导航在多次切换主要页面时稳定显示正文内容，不因异步请求、权限判断或组件异常进入空白状态。
- 通知列表点击通知后可进入稳定详情视图，展示通知标题、内容/描述、状态、时间等信息；不存在通知时给出可理解的空状态或返回入口。
- 更新 E2E 与测试报告，新增缺陷回归用例并记录新鲜验证命令与结果。

**Non-Goals:**

- 不重构整套移动端 UI 或替换现有 Vue Router/状态管理方式。
- 不改变已有 API 前缀 `/api/hx-meeting/v1` 和既有预约、通知、管理端接口语义。
- 不扩大修订范围到平台主应用 `apps/web`、`apps/server` 或 `.claude/skills/`。
- 不把显式 demo 模式作为完成前验证依据。

## Decisions

1. **会议室新增优先走管理端真实接口，而不是只更新前端 demoState。**
   - 做法：检查并补齐 `POST /api/hx-meeting/v1/admin/rooms`；前端 RoomAdmin 页面提供新增表单/抽屉/内联卡片，提交后调用真实 API，并刷新 `GET /admin/rooms`。
   - 理由：用户明确要求“改成真实可用”，完成前验证也要求真实后端联调。
   - 替代方案：仅在前端列表中插入本地数据。该方案无法证明后端持久化，不采用。

2. **底部导航修复以路由与页面容错为核心，避免导航后组件异常导致整页正文丢失。**
   - 做法：梳理 `BottomNavigation.vue`、`AppShell.vue`、`router/index.js` 与主要页面的异步加载逻辑；确保导航目标路由存在、权限受限页面有明确提示或重定向、异步请求失败不阻断主体渲染。
   - 理由：反馈是“所有页面都有概率空白”，通常与路由目标、异步状态、错误未捕获或条件渲染门控有关，需要在公共导航与页面入口层修复。
   - 替代方案：在 E2E 中增加等待时间。该方案不能解决真实用户白屏，不采用。

3. **通知详情采用显式路由与真实数据读取/回退，不在列表点击中直接访问未定义字段。**
   - 做法：增加或修复 `/notifications/:id` 路由和 `NotificationDetailView`（如现有页面缺失则新增），详情数据优先调用真实后端 `GET /notifications/{id}`；若后端暂缺该接口，则在后端补齐，或在前端通过真实列表数据查找并提供非白屏回退。
   - 理由：白屏报错通常来自路由缺失、组件缺失或详情字段为空；显式详情页可被 E2E 稳定覆盖。
   - 替代方案：点击通知只标记已读不跳转。该方案不符合“进入详情页”的用户预期，不采用。

4. **测试新增缺陷回归路径，并保留真实联调断言。**
   - 做法：在 `docs/移动端应用：会议室预约系统/测试/e2e/dev-integration.spec.js` 新增/扩展用例：管理员新增会议室、底部导航循环切换、通知详情点击；更新测试用例与报告。
   - 理由：当前报告未覆盖这些缺陷，修复后需有可复跑证据。
   - 替代方案：只手工记录验证。该方案可追溯性弱，不采用。

## Risks / Trade-offs

- **[风险] 后端缺少会议室新增或通知详情接口，导致前端无法真实提交/读取。** → 在允许范围内补齐生成后端最小接口，并复用现有鉴权、响应 envelope、H2 持久化模式。
- **[风险] 底部导航白屏为偶发问题，单次测试不易复现。** → E2E 采用多轮循环导航与页面关键正文断言，提高回归覆盖强度。
- **[风险] 管理端新增会议室可能受管理员 token 限制。** → E2E 使用 `?token=admin` 或 `admin-token` 验证，普通用户路径不新增管理权限。
- **[风险] 通知列表可能为空导致详情测试无数据。** → 测试前通过真实后端接口读取已有通知；如数据初始化不稳定，补齐后端种子数据或创建可验证通知数据。
- **[风险] 修改 generated 目录可能与重新生成产物冲突。** → 保持变更小而明确，并在测试报告中记录修订点，便于后续再生成时迁移。

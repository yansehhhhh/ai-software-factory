## Context

本次变更是需求产物生成阶段的文档修订，不是运行时代码实现。当前产物位于 `docs/公司内部使用的会议室预约系统/产品设计/`，已覆盖 PRD、版本说明、UI 设计规范、业务流程图、信息架构图、页面流转图和附录。用户反馈集中在业务规则未定稿、术语状态语义混用、页面结构表达不一致，以及 PRD 与版本说明口径不一致。

约束条件：
- 允许修改范围仅限 `docs/公司内部使用的会议室预约系统/产品设计/`。
- 不新增平台运行时能力，不修改 `apps/`、`packages/`、`.claude/` 等目录。
- 修订应保持现有文档结构，做小而明确的文本和流程图调整。
- `PRD.docx` 应与 `PRD.md` 的修订后内容保持一致。

## Goals / Non-Goals

**Goals:**
- 将用户反馈逐项落到产品设计产物中，消除待确认项和歧义表达。
- 统一 PRD、版本说明、UI 设计规范、流程图、页面流转图、信息架构图、术语表中的业务口径。
- 明确可验收规则：转让超时、周期特殊资源整组审批、容量超容提示、二维码展示/下载/打印、取消/释放/自动释放边界、预约时间边界、筛选范围对齐。
- 保持文档可作为后续 UI、研发和测试输入。

**Non-Goals:**
- 不实现前端、后端、数据库或接口代码。
- 不扩展首版范围外能力，例如 PC 后台、规则配置、统计报表、动态二维码等。
- 不重写整套产品文档，只进行针对性修订。

## Decisions

1. **以 PRD 作为业务规则主源，其他产物同步引用同一口径**
   - 决策：先更新 PRD 中功能需求、验收标准、约束与待确认问题，再同步版本说明、UI 规范、流程图和附录。
   - 理由：PRD 是正式需求主文档，其他产物应服务于同一业务规则。
   - 替代方案：仅更新版本说明或附录；缺点是核心需求仍存在歧义。

2. **将原待确认事项转为明确规则**
   - 决策：转让超时、周期特殊资源审批、容量超容、二维码管理方式不再保留为待确认问题，而在对应章节写成确定规则。
   - 理由：用户反馈已明确产品决策，继续保留为待确认会阻碍后续实施。
   - 替代方案：在待确认问题中追加说明；缺点是验收标准不可执行。

3. **预约状态与动作边界分层表达**
   - 决策：取消、主动释放、自动释放分别定义：取消是会前撤销；主动释放是预约人主动放弃使用并释放时间段；自动释放是系统因未签到触发的处理，并单独标识为自动释放/未签到自动释放状态或原因。
   - 理由：避免“已释放”同时承载用户主动行为和系统处罚行为，便于审计、通知和违规统计。
   - 替代方案：继续统一使用“已释放”；缺点是责任归因不清。

4. **容量超出采用允许提交但强提示**
   - 决策：参会人数超过容量时允许提交预约，但在列表、确认页和提交前展示容量风险提示。
   - 理由：会议室容量是建议承载能力，用户反馈要求允许超容但提示。
   - 替代方案：强制阻止提交；与用户反馈不一致。

5. **周期预约特殊资源采用整组审批**
   - 决策：周期预约包含特殊资源时生成整组审批事项，管理员按整组通过或拒绝；拒绝后取消整组关联预约。
   - 理由：降低管理员处理成本，并符合用户明确反馈“整组审批”。
   - 替代方案：逐实例审批；处理成本高且状态复杂。

6. **二维码能力限定在管理员会议室管理页面**
   - 决策：管理员在会议室编辑/二维码页展示二维码，并提供下载、打印入口。
   - 理由：符合固定二维码线下张贴和维护场景，同时不引入动态二维码。
   - 替代方案：员工端展示二维码；与门口固定二维码签到场景不匹配。

7. **预约时间边界采用左闭右开区间**
   - 决策：预约占用与冲突按 `[开始时间, 结束时间)` 判断；开始时间必须早于结束时间，24:00 只作为结束时间。
   - 理由：可避免相邻预约在结束点/开始点被误判冲突，便于后续研发实现。
   - 替代方案：闭区间；会导致相邻时间段边界冲突。

## Risks / Trade-offs

- **PRD.docx 与 PRD.md 不一致** → 修订完成后需重新从 PRD.md 导出或同步更新 docx。
- **流程图与文字规则不一致** → 业务流程图、页面流转图、信息架构图需同步体现整组审批、自动释放、二维码下载/打印等关键变化。
- **状态枚举调整影响后续实现理解** → 在术语表和状态枚举中明确状态含义与触发动作，避免研发自行推断。
- **范围蔓延到运行时代码** → 本 change 的任务仅修改产品设计目录，不触碰应用代码。

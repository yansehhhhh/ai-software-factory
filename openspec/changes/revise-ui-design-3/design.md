## Context

本次修订面向 HX-Meeting 的 UI 设计阶段产物，重点是预约页面的信息架构与交互方式。现有预约页需要从“填写型表单优先”调整为“会议室占用看板优先”：用户先在看板中选择会议室与连续时间段，再在下方表单补充会议主题、参会人等信息。

约束范围仅限 UI 原型目录与 UI 设计规范文件，不涉及前后端实现、接口、数据库或权限模型变更。设计表达需能指导后续实现：既要在视觉稿中体现看板结构，也要在交互说明中明确可选、禁用、已选、冲突等状态。

## Goals / Non-Goals

**Goals:**

- 将预约页主区域设计为会议室占用看板，支持按会议室行和时间格列浏览占用情况。
- 明确看板选择规则：同一行、连续格、最多 4 格（2 小时）、不可跨行、不可跳格、不可跨占用块。
- 将会议室和时间段选择从表单字段前置为看板交互结果，表单只承担信息补充。
- 将参会人字段设计为多选框控件，适配多人参会选择。
- 删除预约页文字版预约规则提示，使用视觉状态、禁用态和错误状态表达规则。
- 同步更新桌面端、移动端、组件库、交互说明、响应式说明和 UI 设计规范。

**Non-Goals:**

- 不修改业务规则本身，例如最小粒度仍按现有半小时格表达。
- 不增加新的审批、支付、签到或会议服务能力。
- 不调整后端接口、数据库表结构或权限控制。
- 不在本次 OpenSpec propose 阶段直接修改阶段产物文件；实际修改由 apply 阶段执行。

## Decisions

1. **预约页采用“看板在上、表单在下”的布局。**
   - 理由：用户反馈明确要求看板可直接选择会议室和时间段；先选资源再补充信息符合预订流程。
   - 替代方案：保留左右分栏表单与日历。该方案无法突出同一会议室行内连续选择规则，因此不采用。

2. **看板以会议室为行、时间段为列，时间格默认 30 分钟粒度。**
   - 理由：用户提出最多连续 4 格等于 2 小时，可推导单格为 30 分钟；行列结构天然避免跨会议室选择。
   - 替代方案：以日历日程卡片方式呈现。该方案对“不可跳格、不可跨占用块”的选择反馈不够直观。

3. **选择状态通过格子样式表达，文字规则提示从预约页面移除。**
   - 理由：用户明确提出不需要文字提示预约规则；规则应由禁用态、占用块、已选块和错误反馈承载。
   - 替代方案：保留规则说明卡片。该方案与反馈冲突，不采用。

4. **表单仅保留补充信息字段。**
   - 理由：会议室和时间段由看板选择产生，避免重复输入和状态不一致；会议主题、参会人仍需用户补充。
   - 替代方案：表单继续提供会议室和时间选择器。该方案会形成两个入口，增加冲突风险。

5. **参会人使用多选框组件。**
   - 理由：用户明确要求参会人为多选框，适合展示常用联系人或部门成员的多选场景。
   - 替代方案：使用自由输入标签。该方案不符合本次反馈，不作为主方案。

## Risks / Trade-offs

- 看板时间格较多导致移动端横向空间不足 → 移动端采用横向滚动时间轴、会议室信息固定或卡片化展示。
- 移除文字规则后用户可能不理解禁用原因 → 通过占用态、不可选光标/禁用样式、选择失败时的轻量错误提示表达，不恢复大段规则说明。
- 多选框参会人数量较多时界面拥挤 → 设计常用参会人分组、搜索/筛选入口或折叠区域，保持表单简洁。
- 看板选择规则较严格，后续实现容易遗漏边界 → 在交互说明和任务中拆分同一行、连续性、最大格数、占用阻断等验收项。

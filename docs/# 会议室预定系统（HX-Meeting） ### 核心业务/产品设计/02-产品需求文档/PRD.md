# 会议室预定系统（HX-Meeting）- 产品需求文档

## 1. 文档概述

### 1.1 文档目的

本文档定义 HX-Meeting 第一版会议室预定系统的产品目标、用户角色、功能范围、业务规则、非功能要求、验收标准、约束与待确认问题，作为后续 UI 设计、架构设计、研发实现、测试验证和上线评估的输入依据。

### 1.2 适用范围

本文档适用于 HX-Meeting 第一版 MVP。第一版以独立系统方式建设，后续再通过接口、入口、SSO 或其他方式集成到“化学通”平台。

### 1.3 文档边界

PRD 主体仅描述产品需求，不包含完整 UI 规范、PlantUML 图全文、完整术语表、完整用户故事、完整测试场景和样例数据。相关内容拆分至：

- `03-UI设计规范/UI-Design-Spec.md`
- `04-流程图/*.puml`
- `05-附录/术语表.md`

## 2. 产品目标

### 2.1 背景

当前会议室使用存在预定不规范、长时间占用、预约后无人使用、线下协调成本高等问题，导致会议室资源利用率下降，并影响员工会议效率。HX-Meeting 旨在建立规范、透明、可追踪的会议室预约与签到机制。

### 2.2 产品定位

HX-Meeting 是面向企业内部用户的会议室预约与使用治理系统。第一版作为独立响应式 Web 系统建设，提供会议室占用查看、预约、扫码签到、自动释放、违规惩罚、预约转让、系统内通知与会议室基础管理能力。

### 2.3 业务目标

| 目标编号 | 目标 | 衡量方式 |
|---|---|---|
| G-001 | 提升会议室预约规范性 | 所有会议室预约均通过系统创建、取消、转让或签到 |
| G-002 | 减少“占而不用” | 未签到预约可被系统自动释放，并形成违规记录 |
| G-003 | 降低协调成本 | 用户可查看会议室占用状态及预约人姓名、部门 |
| G-004 | 提高资源周转效率 | 释放后的剩余时段可被其他用户立即预约 |
| G-005 | 支撑后续化学通集成 | 用户数据、身份字段和日程集成点预留扩展空间 |

## 3. 用户角色

| 角色 | 描述 | 核心权限 |
|---|---|---|
| 普通用户 | 企业内部员工，可预约会议室或作为参会人参与会议 | 查看占用、创建预约、取消本人预约、发起转让、确认转让、扫码签到、查看我的预约/参会/通知 |
| 会议负责人 | 当前预约责任人，可能是原预约人，也可能是转让确认后的接收人 | 对会议签到、释放、转让和违规责任负责 |
| 参会人 | 被加入会议的参与人员 | 查看本人参会记录，在签到窗口内扫码签到 |
| 管理员 | 负责维护会议室基础信息和查看管理入口的人员 | 会议室新增、编辑、停用、二维码管理、查看会议室管理入口 |
| 系统任务 | 系统自动执行定时检查与状态更新 | 自动释放未签到预约、记录违规、触发预约禁用、生成系统内通知 |

## 4. 需求范围

### 4.1 第一版范围内

| 模块 | 范围说明 |
|---|---|
| 登录与身份 | 使用本地/模拟账号登录；预留化学通用户 ID、部门、手机号等字段 |
| 首页 | 展示欢迎词、功能入口、待办/通知摘要；管理员展示会议室管理入口 |
| 会议室占用看板 | 支持按日期查看会议室占用情况；普通用户可见预约人姓名和部门 |
| 会议预约 | 选择会议室、日期、开始/结束时间、会议主题、参会人并提交预约 |
| 预约规则校验 | 最多提前 30 天；每天 08:00-22:00；30 分钟粒度；单次不超过 2 小时；不允许冲突 |
| 系统内日程 | 预约成功后在系统内生成主持人预约记录和参会人参会记录 |
| 扫码签到 | 会议负责人或参会人在会议开始前 15 分钟至开始后 15 分钟内扫码签到 |
| 自动释放 | 会议开始后 15 分钟仍无人签到时，释放剩余时段并记录违规 |
| 违规惩罚 | 自然月内累计 3 次违规后，禁用用户 7 天预约权限 |
| 预约释放 | 当前会议负责人可主动取消未开始预约，释放会议室 |
| 预约转让 | 当前会议负责人发起转让，接收人确认后转让生效并转移责任 |
| 系统内通知/待办 | 预约成功、签到提醒、转让待确认、违规、禁用等事件形成系统内通知 |
| 会议室管理 | 管理员维护会议室名称、位置、容量、二维码、启用/停用状态 |
| 响应式 Web | PC 适配占用看板和管理操作；移动端适配扫码签到、我的预约和通知 |

### 4.2 第一版范围外

| 能力 | 说明 |
|---|---|
| 化学通真实 SSO | 第一版仅预留字段，不真实接入 |
| 化学通日程写入 | 第一版仅系统内日程，不写入化学通日程 |
| 从化学通建日程时预约会议室 | 后续集成阶段评估 |
| 短信、邮件、化学通消息通知 | 第一版仅系统内通知 |
| 设备标签筛选 | 第一版不支持投影仪、白板等标签 |
| 单会议室独立开放时间 | 第一版统一每天 08:00-22:00 |
| 周期性预约 | 第一版不支持每日/每周重复预约 |
| 完整资源运营统计 | 第一版不包含使用统计、排序权重、维护时段等运营能力 |

## 5. 功能需求

### 5.1 登录与身份

#### 功能描述

第一版提供本地/模拟账号登录，用于支持不同角色体验系统功能。用户身份模型需预留后续接入化学通 SSO 所需字段。

#### 业务规则

- 用户至少包含姓名、部门、角色、手机号或工号等基础字段。
- 用户数据需预留化学通用户 ID 字段。
- 管理员权限由系统内角色控制。
- 被禁用预约权限的用户仍可登录、查看、签到、确认转让和接收通知，但不可创建新预约或接收转让成为负责人。

#### 验收标准

- Given 用户使用有效模拟账号，When 提交登录，Then 系统进入首页并展示“欢迎您，{姓名}”。
- Given 用户为管理员，When 进入首页，Then 可看到“会议室管理”入口。
- Given 用户为普通用户，When 进入首页，Then 不展示“会议室管理”入口。

### 5.2 首页与功能入口

#### 功能描述

首页提供用户欢迎信息、核心功能入口和待办/通知摘要，帮助用户快速进入会议预约、会议核销、我的预约等场景。

#### 业务规则

- 所有登录用户可见：会议预约、会议核销、我的预约。
- 管理员额外可见：会议室管理。
- 首页需突出待处理事项，例如待确认转让、即将开始且待签到会议、权限禁用提醒。

#### 验收标准

- Given 用户登录成功，When 进入首页，Then 显示欢迎词和基础功能入口。
- Given 用户存在待确认转让，When 进入首页，Then 待办区域展示转让确认事项。

### 5.3 会议室占用查看

#### 功能描述

用户可查看指定日期各会议室在可预约时段内的占用情况，识别空闲时间，并在占用详情中查看预约人姓名和部门以便线下协调。

#### 业务规则

- 默认展示当天会议室占用情况。
- 支持切换未来 30 天内日期。
- 时间轴按 30 分钟粒度展示，范围为 08:00-22:00。
- 普通用户查看占用详情时，仅展示预约人姓名和部门，不展示会议主题、参会人列表等敏感信息。
- 管理员是否可查看更完整详情：待确认。

#### 验收标准

- Given 某会议室在 10:00-11:00 已被预约，When 用户查看该日期占用看板，Then 对应时间段显示为占用。
- Given 用户点击占用时段，When 弹出详情，Then 仅展示预约人姓名和部门。

### 5.4 创建会议预约

#### 功能描述

用户选择会议室、日期、开始时间、结束时间、参会人等信息后创建预约。预约成功后，系统生成主持人预约记录、参会人参会记录和系统内通知。

#### 业务规则

- 预约日期不得早于当前时间。
- 预约日期最多提前 30 天。
- 可预约时间为每天 08:00-22:00。
- 开始时间和结束时间必须按 30 分钟对齐。
- 单次预约时长不得超过 2 小时。
- 结束时间必须晚于开始时间。
- 同一会议室同一时间段不得存在重叠预约。
- 停用会议室不可被预约。
- 被禁用预约权限的用户不可创建预约。
- 第一版不支持周期性预约。

#### 验收标准

- Given 用户选择未来 30 天内空闲会议室和合法时间段，When 提交预约，Then 系统创建预约并进入“待签到/未开始”状态。
- Given 用户选择 2.5 小时时段，When 提交预约，Then 系统拒绝并提示“单次预约不得超过 2 小时”。
- Given 用户选择已被占用时段，When 提交预约，Then 系统拒绝并提示时间冲突。
- Given 用户处于预约权限禁用期，When 提交预约，Then 系统拒绝并提示禁用截止时间。

### 5.5 系统内日程与我的预约

#### 功能描述

预约成功后，系统内生成会议日程。会议负责人可在“我的预约”查看其负责会议；参会人可在“我的参会”或相同页面的参会标签下查看参与会议。

#### 业务规则

- 第一版不写入化学通日程。
- 会议负责人可查看完整预约详情，包括会议主题、会议室、时间、参会人、签到状态、转让状态等。
- 参会人可查看本人参与会议的基本信息和签到入口。
- 已取消、已释放、已结束、违规释放的记录应保留可追溯状态。

#### 验收标准

- Given 用户成功创建预约，When 打开我的预约，Then 可看到该会议记录。
- Given 用户被添加为参会人，When 打开我的参会，Then 可看到该会议记录。

### 5.6 扫码签到与会议核销

#### 功能描述

会议开始前 15 分钟至开始后 15 分钟内，当前会议负责人或任一参会人扫描会议室二维码完成签到。签到成功后会议室继续保留至预约结束。

#### 业务规则

- 签到二维码与会议室绑定。
- 签到窗口为会议开始前 15 分钟至会议开始后 15 分钟。
- 签到人必须是当前会议负责人或该会议参会人。
- 扫描错误会议室二维码不可签到成功。
- 预约已取消、已释放、已结束时不可签到。
- 签到成功后，预约状态变更为已签到/进行中。
- 只需一次有效签到即可完成核销。

#### 验收标准

- Given 当前时间在签到窗口内且用户为参会人，When 扫描正确会议室二维码，Then 签到成功。
- Given 当前时间早于签到窗口，When 用户扫码，Then 系统提示未到签到时间。
- Given 用户扫描非预约会议室二维码，When 提交签到，Then 系统拒绝并提示会议室不匹配。

### 5.7 未签到自动释放与违规惩罚

#### 功能描述

系统定时检查到会议开始后 15 分钟仍无人签到的预约时，自动释放该预约的剩余时段，并为当前会议负责人记录一次违规。自然月累计 3 次违规后，禁用其 7 天预约权限。

#### 业务规则

- 自动释放发生在会议开始后 15 分钟之后。
- 释放范围为自动释放时刻之后的剩余预约时段。
- 释放后的剩余时段允许其他用户立即预约。
- 违规责任归属于当前会议负责人。
- 若转让已确认，责任归新负责人；若转让未确认，责任仍归原负责人。
- 违规次数按自然月统计，每个自然月重新计数。
- 当月累计第 3 次违规触发 7 天预约权限禁用。
- 禁用期结束后自动恢复预约权限。

#### 验收标准

- Given 预约开始后 15 分钟仍无人签到，When 系统任务执行，Then 预约剩余时段被释放且负责人增加 1 次违规。
- Given 用户当月违规达到 3 次，When 第 3 次违规记录生成，Then 系统禁用其预约权限 7 天。
- Given 用户处于禁用期，When 尝试创建预约，Then 系统拒绝。

### 5.8 主动释放预约

#### 功能描述

当前会议负责人可主动取消未开始或尚未结束的预约，以释放会议室资源。

#### 业务规则

- 当前会议负责人可释放本人负责的预约。
- 已结束预约不可释放。
- 主动释放不计违规。
- 释放成功后，该时段重新可预约。
- 系统需向参会人生成系统内通知。

#### 验收标准

- Given 负责人释放未开始预约，When 确认释放，Then 预约状态变更为已取消/已释放且时间段可再次预约。
- Given 非负责人尝试释放预约，When 提交操作，Then 系统拒绝。

### 5.9 预约转让

#### 功能描述

当前会议负责人可将预约转让给其他用户。接收人确认后，转让正式生效，并由接收人承担后续签到、释放和违规责任。

#### 业务规则

- 仅当前会议负责人可发起转让。
- 转让接收人需为系统有效用户。
- 接收人确认前，原负责人仍承担责任。
- 接收人确认后，负责人变更为接收人。
- 处于预约权限禁用期的用户不可接收转让成为负责人。
- 已取消、已释放、已结束预约不可发起转让。
- 转让需生成系统内待办通知。

#### 验收标准

- Given 负责人发起转让，When 接收人未确认，Then 预约负责人保持不变且接收人看到待确认任务。
- Given 接收人确认转让，When 确认成功，Then 预约负责人变更为接收人。
- Given 接收人处于禁用期，When 原负责人发起转让，Then 系统拒绝。

### 5.10 系统内通知/待办

#### 功能描述

系统通过站内通知和待办向用户提示关键事件，包括预约成功、会议即将开始、转让确认、取消/释放、未签到违规、预约权限禁用等。

#### 业务规则

- 第一版不发送短信、邮件或化学通消息。
- 待确认转让应作为待办展示。
- 即将开始且待签到会议应作为提醒展示。
- 通知需保留已读/未读状态。

#### 验收标准

- Given 用户收到转让请求，When 登录系统，Then 首页待办展示转让确认。
- Given 预约被系统自动释放，When 负责人查看通知，Then 可看到违规和释放说明。

### 5.11 会议室管理

#### 功能描述

管理员维护会议室基础信息，支撑预约、展示和扫码签到。

#### 业务规则

- 管理员可新增、编辑、启用、停用会议室。
- 会议室信息包括名称、位置、容量、二维码、状态。
- 停用会议室不可新增预约，但历史预约记录保留。
- 二维码用于绑定会议室并支持签到校验。

#### 验收标准

- Given 管理员新增会议室并启用，When 用户查看预约页面，Then 该会议室可被选择。
- Given 管理员停用会议室，When 用户创建预约，Then 该会议室不可被选择。
- Given 普通用户访问会议室管理，When 请求页面，Then 系统拒绝或不展示入口。

## 6. 非功能需求

### 6.1 性能要求

| 场景 | 要求 |
|---|---|
| 首页加载 | 常规网络下主要内容 3 秒内可用 |
| 会议室占用看板查询 | 单日占用数据查询 2 秒内返回 |
| 创建预约 | 提交后 2 秒内返回成功或失败结果 |
| 扫码签到 | 扫码提交后 2 秒内返回签到结果 |
| 自动释放任务 | 至少每 5 分钟检查一次待释放预约，具体频率由架构设计确定 |

### 6.2 安全要求

- 用户必须登录后访问业务页面。
- 管理员功能需做角色权限校验。
- 普通用户不可查看会议主题、参会人等敏感占用详情。
- 预约创建、转让、释放、签到和自动释放需保留操作记录或状态变更记录。
- 二维码签到需校验会议室、会议、人员和时间窗口，避免仅凭二维码绕过业务规则。

### 6.3 可用性要求

- PC 和移动端均需可用。
- 移动端扫码签到路径应短，扫码后直接进入签到确认/结果页。
- 错误提示需明确说明原因和下一步操作，例如时间冲突、超出 2 小时、禁用期不可预约等。
- 系统需区分空闲、占用、待签到、已签到、已释放、已取消、违规释放等状态。

### 6.4 兼容性要求

- 支持主流现代浏览器。
- 响应式适配 PC 桌面、平板和手机宽度。
- 移动端需支持通过浏览器打开二维码链接完成签到；是否调用原生摄像头能力由前端与集成方式确定。

### 6.5 可扩展要求

- 用户身份字段预留化学通 SSO 集成。
- 日程模型预留未来同步到化学通日程的扩展字段。
- 通知模型预留未来短信、邮件、化学通消息等渠道扩展。

## 7. 验收标准总览

| 编号 | 验收项 | 通过标准 |
|---|---|---|
| AC-001 | 角色入口 | 管理员可见会议室管理入口，普通用户不可见 |
| AC-002 | 预约规则 | 系统正确限制 30 天、08:00-22:00、30 分钟粒度、单次 2 小时、时间冲突 |
| AC-003 | 占用可见性 | 普通用户仅可见占用人姓名和部门 |
| AC-004 | 系统内日程 | 预约成功后负责人和参会人可在对应列表查看 |
| AC-005 | 签到窗口 | 仅会议开始前 15 分钟至开始后 15 分钟可签到 |
| AC-006 | 签到人员 | 负责人或参会人均可完成签到，其他人不可签到 |
| AC-007 | 自动释放 | 未签到预约在开始后 15 分钟后释放剩余时段并记违规 |
| AC-008 | 违规惩罚 | 自然月内 3 次违规触发 7 天预约权限禁用 |
| AC-009 | 主动释放 | 负责人主动释放后时段可再次预约且不记违规 |
| AC-010 | 转让确认 | 接收人确认后负责人和责任归属变更 |
| AC-011 | 站内通知 | 关键业务事件可生成系统内通知或待办 |
| AC-012 | 响应式可用 | PC 可完成预约和管理，移动端可完成签到和查看 |

## 8. 约束与假设

### 8.1 已确认约束

- 第一版为独立系统，后续集成化学通。
- 第一版仅使用模拟/本地账号，预留化学通 SSO 字段。
- 第一版不真实对接化学通日程。
- 第一版不支持周期性预约。
- 第一版仅系统内通知，不依赖外部通知渠道。
- 第一版会议室管理仅维护基础信息。

### 8.2 产品假设

- 企业内部用户具备统一的姓名、部门等基础身份信息。
- 会议室二维码可被张贴或展示在会议室现场。
- 系统具备后台定时任务能力，用于自动释放与权限恢复。
- 用户可接受站内通知作为第一版提醒方式。

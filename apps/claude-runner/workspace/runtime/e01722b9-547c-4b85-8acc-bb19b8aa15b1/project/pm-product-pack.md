# PM 产品产物包

## 1. 产品需求文档 PRD

### 产品背景
用户希望快速拥有一个可展示个人文字内容的静态博客站点，用于承载个人表达、文章归档与内容浏览。第一版不涉及登录、评论、后台管理与数据库，重点是以较低实现复杂度交付一个具有“温暖、文艺、安静、可阅读”气质的个人博客。

### 目标用户
- 博客主人：希望发布个人文字、随笔、生活记录、读书观影感受的个人创作者
- 博客访客：希望快速浏览首页内容、查看文章列表、阅读文章详情的普通读者
- 设计与开发协作方：需要一个边界明确、内容可演示、适合静态站点实现的 MVP 需求输入

### 用户问题
- 个人表达缺少一个轻量、独立、风格统一的展示空间
- 通用模板博客往往风格偏技术化，不符合“温暖文艺”审美目标
- 仅需展示内容，不需要复杂的发布、登录、数据管理能力
- 没有现成文章数据源，希望产品首版可直接内置示例内容用于展示

### 产品目标
1. 交付一个静态个人博客 MVP，包含首页、文章列表页、文章详情页。
2. 通过视觉与文案风格营造温暖文艺氛围，突出阅读体验。
3. 支持桌面端与手机端响应式浏览。
4. 内置 3-5 篇示例文章，保证首版可直接演示与测试。
5. 全站公开可访问，无需登录、无需数据库、无需后台。

### 功能范围

#### 1）首页
首页承担品牌展示、内容导流与气质表达功能，建议包含：
- 顶部导航：站点名称、首页入口、文章列表入口
- Hero 区：一句博客宣言/简介、温暖文艺风格主视觉文案
- 作者简介区：简短自我介绍或博客说明
- 精选文章区：展示 3-5 篇文章卡片中的部分内容
- 页脚：版权信息、简短 Slogan、基础导航

#### 2）文章列表页
文章列表页承担内容归档与浏览入口功能，建议包含：
- 页面标题与简介
- 文章卡片列表
- 每篇文章展示字段：标题、摘要、发布日期、标签、封面/配图（可选）、阅读入口
- 支持按时间顺序展示（默认倒序）
- 在桌面端与手机端具备良好排版体验

#### 3）文章详情页
文章详情页承担沉浸式阅读功能，建议包含：
- 文章标题
- 发布时间
- 标签
- 头图/题图（可选）
- 正文内容
- 返回文章列表入口
- 上一篇/下一篇导航（推荐，但非强制）

#### 4）内容数据
- 首版内置 3-5 篇示例文章
- 数据来源为静态内置数据
- 不依赖数据库
- 不要求从远程接口获取数据

#### 5）响应式适配
- 必须适配桌面端与手机端
- 保证导航、卡片布局、正文宽度、字体层级在小屏设备可用

### 非目标
以下能力不属于第一版范围：
- 登录/注册
- 评论、点赞、收藏、分享统计
- 后台管理系统
- 数据库持久化
- 文章编辑器
- 搜索功能
- 分类筛选、标签筛选
- Markdown 文件动态读取
- 多作者协作
- 国际化/多语言
- SEO 深度优化以外的高级运营能力

### 核心用户流程

#### 流程 1：访客从首页进入阅读
1. 用户进入首页
2. 看到博客名称、简介与精选文章
3. 点击精选文章卡片
4. 进入文章详情页
5. 阅读正文
6. 返回列表或继续浏览其他文章

#### 流程 2：访客从文章列表查阅内容
1. 用户打开文章列表页
2. 浏览文章卡片
3. 根据标题、摘要、标签和发布日期选择文章
4. 点击进入详情页
5. 完成阅读

#### 流程 3：移动端用户阅读
1. 用户通过手机访问站点
2. 首页内容自动适配小屏布局
3. 用户进入列表页查看文章
4. 进入详情页阅读正文
5. 页面在小屏下保持可读性与可点击性

### 信息架构
- 首页 `/`
- 文章列表页 `/posts`
- 文章详情页 `/posts/:slug`

### 内容与风格要求
- 整体气质：温暖、安静、轻柔、文艺
- 视觉建议：米白、奶油、浅棕、灰绿、淡雾粉等低饱和配色
- 排版建议：偏阅读型布局，正文留白充足，避免信息过密
- 文案建议：避免强商业感与技术感，强调日常记录、阅读感、陪伴感
- 图片策略：可使用静态封面图或纯色占位图，不要求复杂插画系统

### 核心页面功能拆解

#### 首页功能点
- 展示博客品牌名与副标题
- 展示站点简介
- 展示精选文章入口
- 导向文章列表页

#### 文章列表页功能点
- 展示全部内置文章
- 每篇文章以卡片形式呈现
- 点击卡片进入详情

#### 文章详情页功能点
- 渲染完整文章内容
- 展示元信息（时间、标签）
- 提供返回路径

### 非功能需求
- 页面加载应轻量，适合静态部署
- 首屏结构清晰，不依赖复杂交互才能完成浏览
- 正文阅读对比度足够，但整体视觉不过度刺眼
- 手机端点击区域合理，避免误触
- 静态数据结构需易于后续扩展更多文章

### MVP 边界
本次 MVP 仅覆盖：
- 1 个首页
- 1 个文章列表页
- 1 个文章详情页模板
- 3-5 篇内置示例文章
- 桌面端与手机端响应式适配
- 静态内容展示与页面跳转

不覆盖：
- 内容管理后台
- 在线发布流程
- 用户体系
- 评论互动
- 外部 CMS 接入
- 搜索、筛选、分页等增强能力

### 验收摘要
- 提供首页、文章列表页、文章详情页三类页面
- 全站无需登录即可访问
- 不依赖数据库
- 内置 3-5 篇可浏览的示例文章
- 桌面端与手机端均可正常使用
- 风格符合“温暖文艺”方向，具备完整浏览闭环

## 2. 用户故事 User Stories

### 用户角色
- 访客读者
- 博客主人（以内容呈现诉求为主，不包含后台操作）

### Epic 列表
1. 博客形象展示
2. 文章内容浏览
3. 文章详情阅读
4. 响应式阅读体验
5. 示例内容演示能力

### 用户故事

#### Epic 1：博客形象展示
1. 作为访客读者，我希望进入首页时能看到博客名称、简介与风格化视觉氛围，以便快速理解这个博客的主题与气质。
2. 作为博客主人，我希望首页能够体现个人表达风格，以便让博客看起来温暖、有辨识度。

#### Epic 2：文章内容浏览
3. 作为访客读者，我希望查看文章列表，以便快速浏览当前站点有哪些文章可读。
4. 作为访客读者，我希望在文章列表卡片中看到标题、摘要、日期与标签，以便判断是否值得点开阅读。

#### Epic 3：文章详情阅读
5. 作为访客读者，我希望点击文章后进入详情页阅读全文，以便获得完整阅读体验。
6. 作为访客读者，我希望在详情页中看到清晰的标题、时间与正文排版，以便舒适阅读。
7. 作为访客读者，我希望可以从详情页返回文章列表，以便继续浏览其他文章。

#### Epic 4：响应式阅读体验
8. 作为手机端访客，我希望页面在手机上也能正常浏览和点击，以便随时访问博客。
9. 作为桌面端访客，我希望页面在大屏上有舒适留白与合理布局，以便长时间阅读不疲劳。

#### Epic 5：示例内容演示能力
10. 作为产品评审者，我希望系统内置 3-5 篇示例文章，以便在没有后台和数据库的前提下也能直接演示完整站点。
11. 作为开发者，我希望示例数据结构清晰稳定，以便快速渲染首页、列表页和详情页。

### 验收条件

#### Story 1：首页展示博客形象
- Given 用户首次访问博客首页
- When 页面加载完成
- Then 用户应能看到博客名称、简介文案和至少一个明显的内容入口

#### Story 2：首页导流到文章内容
- Given 用户位于首页
- When 用户点击精选文章或“查看文章���入口
- Then 用户应进入文章详情页或文章列表页

#### Story 3：文章列表展示完整摘要信息
- Given 用户进入文章列表页
- When 页面加载完成
- Then 页面应展示全部内置文章卡片
- And 每个卡片至少包含标题、摘要、发布日期、标签、进入详情入口

#### Story 4：文章详情可正常阅读
- Given 用户从首页或文章列表点击某篇文章
- When 进入文章详情页
- Then 页面应正确显示对应文章标题、时间、标签与正文内容

#### Story 5：无登录访问
- Given 任意访客打开站点任一页面
- When 页面加载
- Then 不应出现登录要求、权限拦截或身份验证流程

#### Story 6：静态数据驱动
- Given 系统不接入数据库
- When 页面渲染文章内容
- Then 所有文章内容应来自内置静态数据

#### Story 7：移动端适配
- Given 用户使用手机访问首页、列表页或详情页
- When 页面显示
- Then 文本不应溢出屏幕
- And 主要点击区域应可操作
- And 页面应不依赖横向滚动完成主要浏览

### 优先级建议
- P0
  - 首页基础展示
  - 文章列表页
  - 文章详情页
  - 3-5 篇内置示例文章
  - 桌面端/手机端响应式适配
- P1
  - 首页精选文章区
  - 详情页上一篇/下一篇导航
  - 封面图/题图表现
- P2
  - 更丰富的标签展示
  - 更强的动效或装饰性微交互

## 3. WWAS 条目

### 条目 1：首页
- **Why**：首页是博客第一印象页面，需要同时表达气质与导流内容。
- **What**：交付一个包含站点标题、简介、精选文章与基础导航的首页。
- **Acceptance**：打开首页可看到博客名称、简介、内容入口；可跳转到文章列表或详情。
- **Scope**：视觉氛围、品牌文案、精选内容展示、基础导航。
- **Out of Scope**：轮播广告、登录入口、动态推荐算法。

### 条目 2：文章列表页
- **Why**：访客需要快速浏览全部文章并选择感兴趣的内容。
- **What**：交付一个展示全部内置文章卡片的列表页面。
- **Acceptance**：列表页能展示全部文章；每篇卡片包含标题、摘要、时间、标签和详情入口。
- **Scope**：文章列表渲染、卡片布局、内容跳转。
- **Out of Scope**：搜索、筛选、分页、排序切换。

### 条目 3：文章详情页
- **Why**：文章详情页是核心阅读场景，决定用户是否愿意停留。
- **What**：交付一个可承载完整文章内容的详情页。
- **Acceptance**：进入任一文章详情后，正文内容与元信息完整显示；支持返回其他内容页。
- **Scope**：标题、日期、标签、正文、返回路径、可选上下篇导航。
- **Out of Scope**：评论、点赞、打赏、目录锚点、高亮批注。

### 条目 4：示例文章数据
- **Why**：没有数据库与后台时，需要示例数据支撑完整演示与测试。
- **What**：交付 3-5 篇内置示例文章数据。
- **Acceptance**：首页、列表页与详情页均能消费这些数据完成展示；文章内容互相区分明显。
- **Scope**：文章标题、slug、摘要、正文、日期、标签、封面字段。
- **Out of Scope**：在线编辑、远程同步、本地 Markdown 解析。

### 条目 5：响应式体验
- **Why**：用户已明确需要桌面端和手机端适配。
- **What**：交付可在桌面与手机设备上正常浏览的响应式布局。
- **Acceptance**：三类核心页面在常见桌面与手机宽度下均无关键内容遮挡、溢出与不可点击问题。
- **Scope**：布局切换、字体层级、卡片换行、正文宽度、导航适配。
- **Out of Scope**：平板专项布局优化、原生 App 体验一致性。

### 条目 6：静态站点边界
- **Why**：用户明确不需要登录和数据库，需要保持实现轻量。
- **What**：交付一个纯静态内容展示博客。
- **Acceptance**：站点核心能力不依赖用户身份和数据库即可运行。
- **Scope**：静态页面、静态数据、公开访问。
- **Out of Scope**：服务端鉴权、数据库建模、后台接口。

## 4. 测试场景 Test Scenarios

### 测试目标
验证该静态个人博客是否满足以下目标：
- 页面结构完整
- 浏览链路可用
- 示例数据正确渲染
- 响应式适配生效
- 无登录、无数据库依赖
- 页面风格与阅读体验符合预期

### Happy Path
1. 访问首页，看到博客品牌信息、简介与文章入口。
2. 从首页点击精选文章，成功进入对应文章详情页。
3. 从首页点击“查看全部文章”或导航进入文章列表页。
4. 在文章列表页看到全部内置文章卡片。
5. 点击任意文章卡片，进入正确的文章详情页。
6. 在文章详情页完成正文阅读，并可返回列表继续浏览。

### 边界场景
1. **最少文章数量边界**：仅内置 3 篇文章时，首页精选区与列表页仍正常展示。
2. **最多文章数量边界（本期范围内）**：内置 5 篇文章时，列表布局仍正常。
3. **超长标题**：标题较长时，列表卡片与详情页头部不应破版。
4. **超长摘要**：摘要过长时，应具备截断或合理换行策略。
5. **超长正文**：详情页正文很长时，阅读区域仍保持良好间距与可读性。
6. **标签数量边界**：单篇文章 0 个、1 个、多个标签时，展示均正常。
7. **移动端窄屏**：如 375px 宽度下，导航、卡片、正文无横向溢出。

### 异常与错误状态
1. **文章 slug 不存在**：访问不存在的详情路径时，应显示友好提示或 404 页面/状态。
2. **文章缺少封面图**：无封面图时，页面应使用占位样式或直接省略，不影响阅读。
3. **文章摘要为空**：列表页应有兜底展示策略，不导致布局错乱。
4. **文章标签为空**：详情页/列表页不应出现异常占位文本。
5. **日期格式异常**：若样例数据日期格式不规范，前端应有统一格式约束或回退处理。

### 权限与访问场景
1. 首页公开访问，无需登录。
2. 文章列表页公开访问，无需登录。
3. 文章详情页公开访问，无需登录。
4. 任意页面访问过程中不出现登录弹窗、鉴权跳转或权限报错。

### 数据校验场景
1. 首页精选文章数量应不超过内置文章总量。
2. 文章列表页展示数量应与数据源文章数量一致。
3. 每篇文章 slug 必须唯一。
4. 每篇文章至少包含标题、slug、摘要、发布日期、正文。
5. 文章详情页内容需与对应 slug 的数据严格匹配。

### 场景矩阵

| 场景编号 | 页面 | 前置条件 | 操作 | 预期结果 |
| --- | --- | --- | --- | --- |
| TS-01 | 首页 | 站点已部署或本地可访问 | 打开首页 | 显示博客名称、简介、精选内容和导航 |
| TS-02 | 首页→列表 | 首页正常可访问 | 点击文章列表入口 | 跳转至文章列表页 |
| TS-03 | 首页→详情 | 首页有精选文章 | 点击精选文章卡片 | 跳转至对应文章详情页 |
| TS-04 | 列表页 | 已内置文章数据 | 打开文章列表页 | 展示全部文章卡片 |
| TS-05 | 列表→详情 | 列表页展示正常 | 点击任意文章 | 进入正确详情页 |
| TS-06 | 详情页 | 存在有效文章 slug | 打开详情页 | 显示标题、日期、标签、正文 |
| TS-07 | 无权限流程 | 任意页面可访问 | 直接访问 URL | 不需要登录且无拦截 |

| TS-11 | 错误详情 | 不存在的 slug | 打开错误详情路径 | 展示友好错误状态或 404 |
| TS-12 | 数据完整性 | 已加载数据源 | 检查文章字段 | 必填字段齐全，slug 唯一 |

## 5. 样例数据 Dummy Dataset

### 数据实体

#### 实体 1：BlogProfile（博客信息）
用于首页展示博客基础信息。

#### 实体 2：Post（文章）
用于首页精选、文章列表和文章详情渲染。

### 字段定义

#### BlogProfile
| 字段名 | 类型 | 必填 | 说明 | 示例 |
| --- | --- | --- | --- | --- |
| siteTitle | string | 是 | 博客名称 | 山风与页角 |
| siteSubtitle | string | 是 | 副标题/简介 | 记录日常、阅读与慢下来的片刻 |
| heroIntro | string | 是 | 首页主文案 | 在平凡日子里，认真收集微小的光 |
| authorName | string | 否 | 作者名 | 清和 |
| authorBio | string | 否 | 作者简介 | 喜欢书、植物、旧电影和下雨天 |

#### Post
| 字段名 | 类型 | 必填 | 说明 | 示例 |
| --- | --- | --- | --- | --- |
| id | string | 是 | 唯一标识 | post-001 |
| slug | string | 是 | 详情页路由标识，需唯一 | spring-window |
| title | string | 是 | 文章标题 | 春天把窗台变亮了一些 |
| summary | string | 是 | 文章摘要 | 记一次午后整理窗台时，光线与植物带来的安静感受。 |
| publishedAt | string | 是 | 发布日期 | 2026-04-10 |
| tags | string[] | 否 | 标签列表 | ["生活", "植物", "春天"] |
| coverImage | string | 否 | 封面图地址或占位标识 | /images/post-1.jpg |
| content | string | 是 | 正文内容 | 多段文本 |
| featured | boolean | 否 | 是否首页精选 | true |

### Demo 使用说明
- 默认使用 BlogProfile 1 条记录与 Post 5 条正常样例记录进行页面演示。
- 首页精选文章建议读取 `featured = true` 的前 3 篇文章。
- 文章列表页建议按 `publishedAt` 倒序展示全部文章。
- 文章详情页根据 `slug` 匹配对应文章记录。
- 若封面图资源暂未准备，可用纯色块或占位图替代。

### 测试使用说明
- 正常回归测试使用 5 篇正常样例记录。
- UI 边界测试增加“无标签文章”和“超长标题文章”。
- 数据校验测试加入 slug 重复、标题缺失、正文缺失三类无效记录。
- 404/错误页测试可直接访问不存在的 `/posts/not-exist-slug`。

## 6. 待确认问题
1. 博客名称、作者名与首页主文案当前为示例值，是否需要替换为用户真实信息。
2. 首页是否需要单独的作者照片、手写签名或更明显的个人介绍模块。
3. 文章详情页是否必须提供“上一篇/下一篇”导航，当前建议为推荐项而非强制项。
4. 首页与列表页是否需要固定封面图素材；若没有，首版可使用占位图或纯色视觉块。
5. 当前仅明确适配桌面端与手机端，平板端是否需要专项布局优化暂未确认。
6. 首版未纳入搜索、分类与标签筛选，如后续需要应作为下一阶段增强需求。

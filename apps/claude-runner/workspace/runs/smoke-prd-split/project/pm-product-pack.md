# PM 产品产物包

## 1. 产品需求文档 PRD

### 产品背景

待办事项应用是个人任务管理中最基础、最常见的工具形态。用户需要一个简单、轻量、易上手的应用，用于快速记录待办事项、查看任务状态、完成任务以及删除不再需要的任务。

本次需求聚焦第一版 MVP，不追求复杂协作、提醒、标签体系或多端同步，目标是交付一个可被 UI 设计、前端开发、测试用例和自动化验收直接消费的简单待办事项应用。

### 目标用户

- 个人用户
- 学生
- 办公人员
- 希望快速管理日常任务的轻量用户
- 需要演示 CRUD 能力的应用使用者或测试人员

### 用户问题

用户当前面临的问题包括：

- 临时任务容易遗忘。
- 纸质记录或聊天记录不便于维护状态。
- 简单事项不需要复杂项目管理工具。
- 用户希望快速添加、完成、删除任务。
- 用户希望清楚区分“未完成”和“已完成”的任务。

### 产品目标

第一版 MVP 目标：

1. 支持用户创建待办事项。
2. 支持用户查看待办事项列表。
3. 支持用户标记待办事项为已完成或未完成。
4. 支持用户删除待办事项。
5. 支持基本输入校验和空状态展示。
6. 支持简单本地数据持久化或前端运行时状态保存，具体实现由技术阶段决定。

### 功能范围

#### MVP 必须包含

| 功能模块 | 功能说明 | 优先级 |
|---|---|---|
| 添加待办 | 用户输入任务标题后创建待办事项 | P0 |
| 查看列表 | 展示所有待办事项 | P0 |
| 完成状态切换 | 支持将任务标记为已完成或恢复为未完成 | P0 |
| 删除待办 | 支持删除单条待办事项 | P0 |
| 输入校验 | 空标题不可提交，超长标题给出提示或限制 | P0 |
| 空状态 | 无待办事项时展示友好提示 | P1 |
| 基础统计 | 展示全部、未完成、已完成数量 | P1 |
| 简单筛选 | 支持按全部、未完成、已完成查看 | P1 |

#### 推荐页面结构

| 页面 | 说明 |
|---|---|
| 待办首页 | 唯一主页面，包含输入区、筛选区、统计区、任务列表和空状态 |

#### 主要状态

| 状态 | 说明 |
|---|---|
| 空列表状态 | 当前没有任何待办事项 |
| 正常列表状态 | 存在一条或多条待办事项 |
| 已完成状态 | 单条任务被标记为完成 |
| 未完成状态 | 单条任务尚未完成 |
| 输入错误状态 | 输入为空或不符合规则 |
| 删除后状态 | 删除任务后列表和统计同步更新 |

### 非目标

第一版不包含以下能力：

- 用户登录、注册、账号体系。
- 多用户协作。
- 云端同步。
- 任务截止时间。
- 任务优先级。
- 任务标签或分类。
- 子任务。
- 拖拽排序。
- 任务搜索。
- 日历视图。
- 推送提醒。
- 管理后台。
- AI 自动生成任务。
- 复杂工作流审批。

### 核心用户流程

#### 流程一：添加待办事项

1. 用户进入待办首页。
2. 用户在输入框中输入任务标题。
3. 用户点击“添加”按钮，或按回车提交。
4. 系统校验输入内容。
5. 如果输入合法，系统创建待办事项。
6. 新任务出现在列表中，默认状态为“未完成”。
7. 统计数量同步更新。

#### 流程二：完成待办事项

1. 用户查看待办列表。
2. 用户点击某条任务前的复选框。
3. 系统将该任务标记为已完成。
4. 已完成任务在视觉上与未完成任务区分，例如文字加删除线、颜色变浅。
5. 已完成数量和未完成数量同步更新。

#### 流程三：恢复未完成

1. 用户查看已完成任务。
2. 用户再次点击该任务的复选框。
3. 系统将任务恢复为未完成。
4. 列表状态和统计同步更新。

#### 流程四：删除待办事项

1. 用户查看待办列表。
2. 用户点击某条任务的删除按钮。
3. 系统删除该任务。
4. 任务从列表中移除。
5. 统计数量同步更新。
6. 如果删除后列表为空，展示空状态。

#### 流程五：筛选任务

1. 用户点击筛选项：全部、未完成、已完成。
2. 系统根据任务状态展示对应列表。
3. 当前筛选项应有明显选中状态。
4. 如果筛选结果为空，展示对应空状态提示。

### 验收摘要

MVP 交付后应满足：

- 用户可以成功添加一条合法待办事项。
- 空输入不能添加待办事项，并出现明确反馈。
- 用户可以查看已添加的待办事项。
- 用户可以切换任务完成状态。
- 用户可以删除任务。
- 任务列表、筛选结果和统计数据保持一致。
- 无数据时展示空状态，不出现页面空白或报错。
- 基础页面在桌面端和移动端可用。
- 页面交互无明显阻塞或崩溃。

## 2. 用户故事 User Stories

### 用户角色

| 角色 | 描述 |

| 普通用户 | 使用应用记录和管理个人待办事项的人 |

### 用户故事

#### US-001 添加待办事项

作为普通用户，我希望输入任务标题并添加待办事项，以便记录我接下来需要完成的事情。

#### US-002 查看待办列表

作为普通用户，我希望看到我已经添加的待办事项列表，以便了解当前有哪些任务需要处理。

#### US-003 标记任务完成

作为普通用户，我希望将待办事项标记为已完成，以便区分已经完成和仍需处理的任务。

#### US-004 恢复任务为未完成

作为普通用户，我希望可以把已完成任务恢复为未完成，以便修正误操作或重新处理任务。

#### US-005 删除待办事项

作为普通用户，我希望删除不再需要的待办事项，以便保持列表简洁。

#### US-006 查看任务统计

作为普通用户，我希望看到全部、未完成、已完成任务数量，以便快速了解任务概况。

#### US-007 按状态筛选任务

作为普通用户，我希望按全部、未完成、已完成筛选任务，以便聚焦查看特定状态的任务。

#### US-008 空输入提示

作为普通用户，我希望在没有输入任务标题时无法添加任务，并看到提示，以便知道如何正确操作。

#### US-009 空列表提示

作为普通用户，我希望在没有任务时看到友好的空状态提示，以便知道当前没有待办事项。

## 3. WWAS 条目

### WWAS-001 添加待办事项

- Why：用户需要快速记录待处理事项，避免遗漏任务。
- What：提供一个输入框和添加按钮，允许用户创建新的待办事项。
- Acceptance：
  - 输入合法标题后可以成功添加任务。
  - 新任务默认状态为未完成。
  - 添加成功后输入框清空。
  - 空输入不可添加。
- Scope：
  - 单行文本输入。
  - 添加按钮。
  - 回车快捷提交可作为推荐实现。
  - 标题基础校验。
- Out of Scope：
  - 富文本内容。
  - 附件上传。
  - 截止时间。
  - 任务优先级。
  - 多字段创建表单。

### WWAS-002 展示待办列表

- Why：用户需要看到已记录的所有待办事项。
- What：展示待办事项列表，每条任务包含标题、完成状态和删除入口。
- Acceptance：
  - 有任务时展示任务列表。
  - 每条任务标题准确显示。
  - 每条任务可以独立操作。
  - 没有任务时展示空状态。
- Scope：
  - 全部任务列表。
  - 单条任务基础信息。
  - 空状态提示。
- Out of Scope：
  - 分页。
  - 无限滚动。
  - 拖拽排序。
  - 分组展示。

### WWAS-003 切换完成状态

- Why：用户需要追踪哪些任务已完成，哪些任务仍待处理。
- What：提供完成状态切换能力，用户可将任务标记为已完成或恢复未完成。
- Acceptance：
  - 未完成任务点击后变为已完成。
  - 已完成任务点击后变为未完成。
  - 已完成任务有明显视觉区分。
  - 统计数据同步变化。
- Scope：
  - 二元状态：未完成、已完成。
  - 单条任务状态切换。
- Out of Scope：
  - 多阶段任务状态。
  - 审批状态。
  - 进行中状态。
  - 自定义状态。

### WWAS-004 删除待办事项

- Why：用户需要清理已经无效或不再需要的任务。
- What：每条任务提供删除入口，点击后移除该任务。
- Acceptance：
  - 点击删除后任务从列表中消失。
  - 删除后统计数据同步更新。
  - 删除最后一条任务后展示空状态。
- Scope：
  - 删除单条任务。
  - 前端状态更新。
- Out of Scope：
  - 批量删除。
  - 回收站。
  - 撤销删除。
  - 删除确认弹窗，MVP 可选。

### WWAS-005 筛选待办事项

- Why：当任务数量增加时，用户需要聚焦查看特定状态的任务。
- What：提供全部、未完成、已完成三个筛选项。
- Acceptance：
  - 点击“全部”展示所有任务。
  - 点击“未完成”只展示未完成任务。
  - 点击“已完成”只展示已完成任务。
  - 当前筛选项有选中状态。
  - 筛选结果为空时展示对应提示。
- Scope：
  - 状态筛选。
  - 筛选项视觉选中态。
- Out of Scope：
  - 关键词搜索。
  - 标签筛选。
  - 日期筛选。
  - 高级过滤器。

### WWAS-006 任务统计

- Why：用户需要快速了解当前任务总量和完成情况。
- What：展示全部任务数、未完成任务数、已完成任务数。
- Acceptance：
  - 添加任务后全部和未完成数量增加。
  - 完成任务后未完成减少、已完成增加。
  - 恢复任务后未完成增加、已完成减少。
  - 删除任务后相关数量同步减少。
- Scope：
  - 三个基础数字统计。
- Out of Scope：
  - 完成率图表。
  - 历史趋势。
  - 周报/月报。
  - 用户效率分析。

### WWAS-007 输入校验与错误反馈

- Why：系统需要避免创建空白或不合理的任务数据。
- What：对任务标题进行基础校验，并展示错误提示。
- Acceptance：
  - 空字符串不可提交。
  - 仅空格内容不可提交。
  - 超过最大长度时阻止提交或提示用户。
  - 错误提示应清晰可见。
- Scope：
  - 必填校验。
  - 去除首尾空格。
  - 标题长度限制。
- Out of Scope：
  - 敏感词审核。
  - AI 内容检测。
  - 多语言校验。
  - 复杂规则引擎。

## 4. 测试场景 Test Scenarios

### 测试目标

测试目标是验证简单待办事项应用在 MVP 范围内的核心功能、边界条件、异常状态、权限访问和数据校验行为是否符合预期。

重点覆盖：

- 添加待办事项。
- 查看待办列表。
- 标记完成和恢复未完成。
- 删除待办事项。
- 统计数量更新。
- 状态筛选。
- 空状态。
- 输入校验。
- 访客访问。
- 基础响应式可用性。

### Happy Path

#### HP-001 成功添加待办

Given 用户打开待办首页  
When 用户输入“完成日报”并点击“添加”  
Then 页面新增一条“完成日报”任务  
And 任务状态为未完成

Then 该任务显示为已完成  
And 已完成数量增加 1

When 用户再次点击该任务复选框  
Then 该任务显示为未完成  
And 未完成数量增加 1  

#### HP-004 成功删除任务

When 用户点击删除按钮

Given 页面存在 1 条未完成任务和 1 条已完成任务  
When 用户点击“已完成”筛选项  
Then 列表只显示已完成任务  

### 边界场景

#### B-001 添加最短有效标题

- 输入：`A`
- 预期：允许添加。

#### B-002 添加最大长度标题

- 输入：长度等于最大限制的标题，例如 100 个字符。
- 预期：允许添加或根据产品限制正常处理。

#### B-003 输入超过最大长度标题

- 输入：超过最大限制，例如 101 个字符。
- 预期：阻止提交并提示标题过长，或输入框限制继续���入。

#### B-004 删除最后一条任务

- 前置：列表只有 1 条任务。
- 操作：删除该任务。
- 预期：列表为空，展示空状态。

#### B-005 筛选结果为空

- 前置：只有未完成任务。
- 操作：点击“已完成”筛选。
- 预期：展示“暂无已完成任务”之类提示。

#### B-006 连续快速添加多个任务

- 操作：快速输入并添加多条不同任务。
- 预期：任务均被正确添加，列表和统计准确。

### 异常与错误状态

#### E-001 空输入提交

- 输入：空字符串。
- 操作：点击添加。
- 预期：不创建任务，展示错误提示。

#### E-002 仅空格提交

- 输入：`   `
- 操作：点击添加。
- 预期：不创建任务，展示错误提示。

#### E-003 重复任务标题

- 输入：与已有任务同名。
- 预期：MVP 可允许重复；如技术实现选择不允许，应提示用户。
- 默认建议：允许重复，避免增加复杂规则。

#### E-004 删除不存在的任务

- 场景：由于状态不同步或重复点击删除导致目标任务不存在。
- 预期：页面不崩溃，列表保持正常。

#### E-005 状态切换失败

- 场景：状态更新操作异常。
- 预期：页面不崩溃，可保持原状态并提示用户，具体取决于实现是否存在后端接口。

### 权限与访问场景

#### P-001 访客直接访问应用

Given 用户未登录  
When 用户打开待办首页  
Then 用户可以正常使用待办功能  

#### P-002 无账号隔离要求

Given MVP 不包含账号系统  
When 多个用户在同一浏览器环境使用  
Then 系统不区分用户身份  

#### P-003 刷新页面后的访问

Given 用户已添加任务  
When 用户刷新页面  
Then 如果实现本地持久化，任务应保留  
Or 如果实现运行时状态，任务可清空，但应符合技术实现说明  

默认建议：MVP 使用本地存储时，刷新后保留任务；若只是演示版本，可接受刷新后重置，但需在实现说明中明确。

### 数据校验场景

#### V-001 标题必填

- 字段：title
- 规则：必填
- 无效值：空字符串、仅空格
- 预期：阻止创建

#### V-002 标题去除首尾空格

- 输入：`  买菜  `
- 预期：保存为 `买菜`

#### V-003 标题长度限制

- 字段：title
- 建议最大长度：100 字符
- 超过后：阻止提交或限制输入

#### V-004 状态字段有效性

- 字段：completed
- 有效值：true、false
- 默认值：false

#### V-005 ID 唯一性

- 字段：id
- 规则：每条任务唯一
- 预期：新增任务不会覆盖已有任务

## 6. 待确认问题

1. 第一版是否需要刷新页面后保留待办数据？
   - 默认建议：使用浏览器本地存储保留数据。
   - 如果只是一次性演示，也可以接受刷新后重置。

2. 任务标题最大长度是否确定为 100 字符？
   - 默认建议：100 字符，避免列表布局异常。

3. 是否需要删除确认？
   - 默认建议：MVP 不需要确认弹窗，点击即删除，保持简单。
   - 如面向真实用户，可增加确认或撤销能力。

4. 是否允许重复标题？
   - 默认建议：允许重复，减少创建限制。

5. 是否需要一键清空已完成任务？
   - 默认建议：不纳入 MVP，可作为后续增强。

6. 是否需要支持移动端适配？
   - 默认建议：需要基础响应式布局，保证手机浏览器可用。

7. 是否需要后端 API？
   - 默认建议：MVP 可先使用前端本地状态或本地存储实现。
   - 如主链路需要前后端联调，可增加最小 REST API：创建、列表、更新状态、删除。

8. 是否需要用户登录？
   - 默认建议：不需要，MVP 面向单用户轻量使用。

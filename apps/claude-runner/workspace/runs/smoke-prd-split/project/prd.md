# PRD

## 产品需求文档
### 产品背景

待办事项应用是个人任务管理中最基础、最常见的工具形态。用户需要一个简单、轻量、易上手的应用，用于快速记录待办事项、查看任务状态、完成任务以及删除不再需要的任务。

本次需求聚焦第一版 MVP，不追求复杂协作、提醒、标签体系或多端同步，目标是交付一个可被 UI 设计、前端开发、测试用例和自动化验收直接消费的简单待办事项应用。

### 目标用户

- 个人用户
- 学生
- 办公人员
- 希望快速管理日常任务的轻量用户
- 需要演示 CRUD 能力的应用使用者或测试人员

### 用户问题

用户当前面临的问题包括：

- 临时任务容易遗忘。
- 纸质记录或聊天记录不便于维护状态。
- 简单事项不需要复杂项目管理工具。
- 用户希望快速添加、完成、删除任务。
- 用户希望清楚区分“未完成”和“已完成”的任务。

### 产品目标

第一版 MVP 目标：

1. 支持用户创建待办事项。
2. 支持用户查看待办事项列表。
3. 支持用户标记待办事项为已完成或未完成。
4. 支持用户删除待办事项。
5. 支持基本输入校验和空状态展示。
6. 支持简单本地数据持久化或前端运行时状态保存，具体实现由技术阶段决定。

### 功能范围

#### MVP 必须包含

| 功能模块 | 功能说明 | 优先级 |
|---|---|---|
| 添加待办 | 用户输入任务标题后创建待办事项 | P0 |
| 查看列表 | 展示所有待办事项 | P0 |
| 完成状态切换 | 支持将任务标记为已完成或恢复为未完成 | P0 |
| 删除待办 | 支持删除单条待办事项 | P0 |
| 输入校验 | 空标题不可提交，超长标题给出提示或限制 | P0 |
| 空状态 | 无待办事项时展示友好提示 | P1 |
| 基础统计 | 展示全部、未完成、已完成数量 | P1 |
| 简单筛选 | 支持按全部、未完成、已完成查看 | P1 |

#### 推荐页面结构

| 页面 | 说明 |
|---|---|
| 待办首页 | 唯一主页面，包含输入区、筛选区、统计区、任务列表和空状态 |

#### 主要状态

| 状态 | 说明 |
|---|---|
| 空列表状态 | 当前没有任何待办事项 |
| 正常列表状态 | 存在一条或多条待办事项 |
| 已完成状态 | 单条任务被标记为完成 |
| 未完成状态 | 单条任务尚未完成 |
| 输入错误状态 | 输入为空或不符合规则 |
| 删除后状态 | 删除任务后列表和统计同步更新 |

### 非目标

第一版不包含以下能力：

- 用户登录、注册、账号体系。
- 多用户协作。
- 云端同步。
- 任务截止时间。
- 任务优先级。
- 任务标签或分类。
- 子任务。
- 拖拽排序。
- 任务搜索。
- 日历视图。
- 推送提醒。
- 管理后台。
- AI 自动生成任务。
- 复杂工作流审批。

### 核心用户流程

#### 流程一：添加待办事项

1. 用户进入待办首页。
2. 用户在输入框中输入任务标题。
3. 用户点击“添加”按钮，或按回车提交。
4. 系统校验输入内容。
5. 如果输入合法，系统创建待办事项。
6. 新任务出现在列表中，默认状态为“未完成”。
7. 统计数量同步更新。

#### 流程二：完成待办事项

1. 用户查看待办列表。
2. 用户点击某条任务前的复选框。
3. 系统将该任务标记为已完成。
4. 已完成任务在视觉上与未完成任务区分，例如文字加删除线、颜色变浅。
5. 已完成数量和未完成数量同步更新。

#### 流程三：恢复未完成

1. 用户查看已完成任务。
2. 用户再次点击该任务的复选框。
3. 系统将任务恢复为未完成。
4. 列表状态和统计同步更新。

#### 流程四：删除待办事项

1. 用户查看待办列表。
2. 用户点击某条任务的删除按钮。
3. 系统删除该任务。
4. 任务从列表中移除。
5. 统计数量同步更新。
6. 如果删除后列表为空，展示空状态。

#### 流程五：筛选任务

1. 用户点击筛选项：全部、未完成、已完成。
2. 系统根据任务状态展示对应列表。
3. 当前筛选项应有明显选中状态。
4. 如果筛选结果为空，展示对应空状态提示。

### 验收摘要

MVP 交付后应满足：

- 用户可以成功添加一条合法待办事项。
- 空输入不能添加待办事项，并出现明确反馈。
- 用户可以查看已添加的待办事项。
- 用户可以切换任务完成状态。
- 用户可以删除任务。
- 任务列表、筛选结果和统计数据保持一致。
- 无数据时展示空状态，不出现页面空白或报错。
- 基础页面在桌面端和移动端可用。
- 页面交互无明显阻塞或崩溃。

# PM 产品产物包

## 1. 产品需求文档 PRD

### 产品背景

用户在日常工作、学习和生活中经常需要记录待办事项，并跟踪事项是否完成。第一版产品目标是提供一个简单、轻量、可快速使用的待办事项应用，帮助用户完成“记录任务 → 查看任务 → 标记完成 → 删除任务”的基础闭环。

本产品作为 AI 软件工厂自动化主链路的 MVP 示例应用，重点用于后续 UI 设计、前端代码生成、测试用例生成和端到端验证，不追求复杂协作、账号体系或多端同步能力。

### 目标用户

- 普通个人用户
- 需要快速记录任务的学生、上班族、自由职业者
- 需要一个简单 demo 应用验证 AI 软件工厂生成链路的开发与测试人员

### 用户问题

- 用户容易忘记临时任务或日常事项。
- 用户需要一个无需复杂配置即可使用的任务清单。
- 用户希望快速知道哪些任务已完成，哪些任务仍待处理。
- 在 demo 场景中，开发和测试链路需要一个结构简单但具备基本交互状态的应用。

### 产品目标

第一版 MVP 目标：

- 支持创建待办事项。
- 支持展示待办事项列表。
- 支持将待办事项标记为完成或未完成。
- 支持删除待办事项。
- 支持基础空状态、错误状态和输入校验。
- 支持前端本地数据持久化，刷新页面后任务不丢失。

可衡量目标：

- 用户可以在 10 秒内新增一条待办。
- 用户可以一眼区分已完成和未完成事项。
- 主要功能路径具备可自动化测试的稳定选择器或明确页面状态。
- MVP 不依赖后端服务即可运行。

### 功能范围

#### MVP 功能

1. 新增待办事项
   - 用户可在输入框中输入任务标题。
   - 点击“添加”按钮或按 Enter 后创建任务。
   - 输入为空或全为空格时不允许创建。

2. 查看待办列表
   - 展示全部待办事项。
   - 每条待办展示任务标题、完成状态、删除操作。
   - 支持空列表状态展示。

3. 完成状态切换
   - 用户可将待办标记为已完成。
   - 用户可将已完成待办恢复为未完成。
   - 已完成任务在视觉上与未完成任务区分，例如删除线、灰色文字或完成标签。

4. 删除待办事项
   - 用户可删除任意一条待办。
   - 删除后列表立即更新。

5. 本地持久化
   - 使用浏览器本地存储保存待办事项。
   - 页面刷新后仍保留已有待办。

6. 基础统计
   - 展示总任务数。
   - 展示未完成任务数。
   - 展示已完成任务数。

7. 基础筛选
   - 支持查看全部任务。
   - 支持只查看未完成任务。
   - 支持只查看已完成任务。

### 非目标

第一版不包含：

- 用户注册、登录、权限体系。
- 多用户协作。
- 云端同步。
- 截止日期、提醒、日历集成。
- 优先级、标签、项目分组。
- 拖拽排序。
- 批量编辑。
- 复杂搜索。
- 管理后台。
- AI 自动生成任务。
- 服务端 API 或数据库存储。

### 核心用户流程

#### 流程一：新增待办

1. 用户打开待办应用首页。
2. 页面展示标题、输入框、添加按钮、待办列表区域。
3. 用户在输入框输入任务标题。
4. 用户点击“添加”或按 Enter。
5. 系统校验输入内容。
6. 如果输入有效，系统创建新待办并展示在列表中。
7. 输入框清空，统计数据更新。

#### 流程二：完成待办

1. 用户在待办列表中看到一条未完成任务。
2. 用户点击该任务的复选框或完成按钮。
3. 系统将任务状态更新为已完成。
4. 页面视觉状态更新。
5. 统计数据更新。

#### 流程三：恢复未完成

1. 用户在列表中看到一条已完成任务。
2. 用户点击复选框或完成按钮。
3. 系统将任务状态更新为未完成。
4. 页面视觉状态更新。
5. 统计数据更新。

#### 流程四：删除待办

1. 用户在列表中选择一条待办。
2. 用户点击删除按钮。
3. 系统删除该待办。
4. 列表和统计数据立即更新。
5. 如果删除后列表为空，展示空状态。

#### 流程五：筛选待办

1. 用户点击“全部”“未完成”“已完成”中的任一筛选项。
2. 系统根据当前筛选条件展示对应任务。
3. 统计数据保持展示全局统计，列表展示当前筛选结果。

### 验收摘要

MVP 验收通过条件：

- 用户可以成功新增非空待办事项。
- 空输入不能新增待办，并展示明确提示或保持不可提交状态。
- 待办事项可以在未完成和已完成之间切换。
- 待办事项可以被删除。
- 空列表时有明确空状态。
- 筛选功能可以正确展示全部、未完成、已完成任务。
- 页面刷新后，已创建任务仍然存在。
- 基础统计数量正确。
- 应用在无后端环境下可运行。
- 核心流程可被自动化测试覆盖。

### 用户故事

#### US-001 新增待办

作为普通用户，我希望输入任务标题并添加待办，以便记录我接下来需要完成的事情。

#### US-002 阻止空待办

作为普通用户，我希望系统阻止我添加空白待办，以便保持列表内容有效。

#### US-003 查看待办列表

作为普通用户，我希望看到所有已添加的待办事项，以便了解当前任务清单。

#### US-004 查看空状态

作为普通用户，我希望在没有待办时看到空状态提示，以便知道当前暂无任务并可以开始添加。

#### US-005 标记完成

作为普通用户，我希望将待办标记为已完成，以便区分已完成任务和未完成任务。

#### US-006 恢复未完成

作为普通用户，我希望将已完成待办恢复为未完成，以便修正误操作或重新跟进任务。

#### US-007 删除待办

作为普通用户，我希望删除不再需要的待办，以便保持任务列表简洁。

#### US-008 按状态筛选

作为普通用户，我希望按全部、未完成、已完成筛选待办，以便快速查看特定状态的任务。

#### US-009 查看统计

作为普通用户，我希望看到总任务数、未完成数和已完成数，以便快速了解任务进度。

#### US-010 刷新后保留数据

作为普通用户，我希望刷新页面后任务仍然存在，以便不会因为页面刷新丢失记录。

## 3. WWAS 条目

### WWAS-001 新增待办

- Why：用户需要快速记录即将处理的任务，形成待办清单。
- What：提供任务输入框和添加按钮，支持输入有效标题后创建待办。
- Acceptance：
  - 输入非空标题后可以新增待办。
  - 新增待办默认状态为未完成。
  - 新增成功后输入框清空。
  - 新增后列表和统计数据同步更新。
- Scope：
  - 单行文本标题。
  - 点击按钮添加。
  - 按 Enter 添加。
  - 基础输入校验。
- Out of Scope：
  - 富文本任务内容。
  - 截止日期。
  - 优先级。
  - 附件。
  - 多人分配。

### WWAS-002 输入校验

- Why：避免创建无意义的空白任务，提升数据质量。
- What：对待办标题进行基础校验，阻止空字符串和纯空格提交。
- Acceptance：
  - 空输入不能提交。
  - 纯空格不能提交。
  - 提交无效输入时不会新增列表项。
  - 用户能理解为什么无法添加，例如按钮禁用或展示错误提示。
- Scope：
  - 前端输入校验。
  - 标题前后空格自动裁剪。
- Out of Scope：
  - 敏感词审核。
  - 重复任务强制拦截。
  - 服务端校验。

### WWAS-003 待办列表展示

- Why：用户需要查看所有已记录任务，判断后续行动。
- What：展示待办事项列表，每条包含标题、完成状态和删除操作。
- Acceptance：
  - 列表能展示所有当前筛选条件下的待办。
  - 每条待办标题清晰可读。
  - 已完成和未完成状态有视觉差异。
  - 列表为空时展示空状态。
- Scope：
  - 列表展示。
  - 空状态展示。
  - 基础响应式布局。
- Out of Scope：
  - 分页。
  - 无限滚动。
  - 拖拽排序。
  - 分组展示。

### WWAS-004 完成状态切换

- Why：用户需要追踪任务进度，知道哪些任务已经完成。
- What：提供状态切换控件，支持未完成和已完成之间切换。
- Acceptance：
  - 未完成任务可切换为已完成。
  - 已完成任务可切换为未完成。
  - 状态切换后视觉样式立即更新。
  - 统计数据立即更新。
  - 刷新后状态保持。
- Scope：
  - 单条任务状态切换。
  - 本地状态保存。
- Out of Scope：
  - 批量完成。
  - 任务完成时间展示。
  - 完成历史记录。

### WWAS-005 删除待办

- Why：用户需要移除不再需要或误创建的任务，保持列表干净。
- What：每条待办提供删除按钮，点击后删除该任务。
- Acceptance：
  - 用户可删除任意一条待办。
  - 删除后该任务不再显示。
  - 统计数据同步更新。
  - 删除后刷新页面，该任务不会恢复。
- Scope：
  - 单条删除。
  - 删除后立即更新本地存储。
- Out of Scope：
  - 批量删除。
  - 撤销删除。
  - 回收站。
  - 删除二次确认。

### WWAS-006 筛选待办

- Why：当任务数量增加后，用户需要快速聚焦未完成或已完成任务。
- What：提供全部、未完成、已完成三个筛选入口。
- Acceptance：
  - 默认展示全部任务。
  - 点击未完成后只展示未完成任务。
  - 点击已完成后只展示已完成任务。
  - 当前筛选项有明确选中状态。
  - 切换筛选不会修改任务数据。
- Scope：
  - 三种固定筛选条件。
  - 当前页面内即时筛选。
- Out of Scope：
  - 搜索。
  - 多条件筛选。
  - 自定义筛选器。

### WWAS-007 统计信息

- Why：用户需要快速了解任务总量和完成进度。
- What：展示总任务数、未完成数、已完成数。
- Acceptance：
  - 新增任务后总数和未完成数增加。
  - 完成任务后已完成数增加、未完成数减少。
  - 恢复未完成后未完成数增加、已完成数减少。
  - 删除任务后相关统计正确更新。
- Scope：
  - 三个基础统计数字。
- Out of Scope：
  - 百分比进度条。
  - 趋势图。
  - 历史统计报表。

### WWAS-008 本地持久化

- Why：用户刷新页面后不应丢失已经记录的任务。
- What：使用浏览器本地存储保存待办数据。
- Acceptance：
  - 新增、删除、状态切换后数据保存到本地。
  - 刷新页面后数据恢复。
  - 本地无数据时展示空状态。
  - 本地数据异常时应用不崩溃，并回退为空列表。
- Scope：
  - 浏览器本地存储。
  - 单设备单浏览器。
- Out of Scope：
  - 跨设备同步。
  - 后端数据库。
  - 用户账号隔离。

## 4. 测试场景 Test Scenarios

### 测试目标

- 验证用户可以完成待办应用的核心任务管理闭环。
- 验证输入校验、状态切换、删除、筛选、统计和持久化行为正确。
- 验证异常数据和空状态不会导致页面崩溃。
- 验证 MVP 在无登录、无后端的情况下可以正常使用。

### Happy Path

#### HP-001 添加一条待办

步骤：

1. 打开应用首页。
2. 在输入框输入“购买牛奶”。
3. 点击“添加”按钮。
4. 查看列表。

预期结果：

- 列表出现“购买牛奶”。
- 任务状态为未完成。
- 输入框清空。
- 总数为 1，未完成数为 1，已完成数为 0。

预期结果：

- 页面仍展示“阅读文档”。
- 状态仍为已完成。

## 6. 待确认问题

以下问题不阻塞第一版 MVP，可在后续版本或实现前进一步确认：

1. 是否要求应用使用特定前端技术栈，例如 Vue 3、React 或原生 HTML/CSS/JS？
2. 是否需要默认预置 demo 数据，还是首次打开必须为空列表？
3. 待办标题最大长度是否确认为 100 个字符？
4. 删除操作是否需要二次确认或撤销能力？
5. 是否需要移动端优先适配，或仅需桌面端基础响应式？
6. 是否需要支持深色模式？
7. 是否需要将筛选状态同步到 URL 参数中？
8. 是否需要在本地存储不可用时展示明确警告？
9. 是否需要支持任务创建时间或更新时间在 UI 上展示？
10. 是否需要支持清空所有已完成任务的快捷操作？

# PM 产品产物包

## 1. 产品需求文档 PRD

### 产品背景

用户需要一个简单的待办事项应用，用于记录、查看、完成和管理日常任务。第一版 MVP 聚焦个人使用场景，帮助用户快速创建待办、标记完成、删除不需要的事项，并通过清晰的列表状态了解当前任务进展。

本产品将作为 AI 软件工厂主链路中的示例应用需求输入，可供后续 UI 设计、前端代码生成和自动化测试阶段消费。

### 目标用户

- 有日常任务记录需求的个人用户
- 希望快速管理待办事项的轻量级工具用户
- 需要一个简单 demo 应用来验证前端生成、状态管理和测试流程的开发/测试人员

### 用户问题

- 用户容易忘记临时任务或日常事项
- 用户需要快速记录任务，而不希望使用复杂项目管理工具
- 用户需要区分“未完成”和“已完成”的事项
- 用户需要删除不再需要的待办事项
- 用户需要在页面刷新后仍能保留本地待办数据

### 产品目标

第一版 MVP 目标：

1. 支持用户创建待办事项
2. 支持展示待办事项列表
3. 支持标记待办事项为完成/未完成
4. 支持删除待办事项
5. 支持按状态筛选全部、未完成、已完成事项
6. 支持展示待办数量统计
7. 使用浏览器本地存储保存数据，刷新页面后数据不丢失
8. 提供清晰的空状态、输入校验和基础错误提示

### 功能范围

#### MVP 必做功能

| 模块 | 功能 | 说明 |
|---|---|---|
| 待办创建 | 新增待办 | 用户输入任务标题后创建待办 |
| 输入校验 | 空内容校验 | 不允许创建空待办 |
| 输入校验 | 长度限制 | 待办标题建议限制在 1-100 个字符 |
| 待办列表 | 查看全部待办 | 默认展示全部待办 |
| 状态管理 | 标记完成 | 用户可将未完成事项标记为已完成 |
| 状态管理 | 取消完成 | 用户可将已完成事项恢复为未完成 |
| 删除管理 | 删除待办 | 用户可删除单条待办 |
| 筛选 | 全部/未完成/已完成 | 用户可切换列表展示范围 |
| 统计 | 数量统计 | 展示全部、未完成、已完成数量 |
| 本地持久化 | localStorage 保存 | 页面刷新后保留数据 |
| 空状态 | 无数据提示 | 无待办或筛选结果为空时展示提示 |

#### 页面范围

第一版仅包含一个主页面：

- 页面名称：待办事项首页
- 页面内容：
  - 应用标题
  - 新增待办输入框
  - 新增按钮
  - 筛选标签/按钮
  - 待办事项列表
  - 任务统计
  - 空状态提示

#### 数据流

1. 用户在输入框输入待办标题
2. 点击新增按钮或按 Enter
3. 前端校验输入内容
4. 校验通过后创建待办对象
5. 待办对象加入本地状态
6. 状态同步写入 localStorage
7. 页面重新渲染列表与统计信息
8. 用户完成、取消完成或删除时，重复更新状态与 localStorage

### 非目标

以下内容不包含在第一版 MVP 中：

- 用户注册、登录、账号体系
- 多用户协作
- 服务端 API 与数据库
- 待办分类、标签、优先级
- 截止日期、提醒通知
- 拖拽排序
- 子任务
- 任务搜索
- 批量操作
- 云端同步
- 多设备同步
- AI 自动生成任务
- 管理后台

### 核心用户流程

#### 流程一：新增待办

1. 用户打开待办事项应用
2. 用户看到输入框和新增按钮
3. 用户输入“买牛奶”
4. 用户点击新增按钮或按 Enter
5. 系统校验输入非空
6. 系统创建一条未完成待办
7. 系统在列表中展示“买牛奶”
8. 系统更新未完成数量

#### 流程二：完成待办

1. 用户看到未完成待办列表
2. 用户点击某条待办前的复选框
3. 系统将该待办状态改为已完成
4. 系统以完成样式展示该待办，例如文字删除线或弱化颜色
5. 系统更新未完成和已完成数量

#### 流程三：取消完成

1. 用户看到已完成待办
2. 用户再次点击该待办复选框
3. 系统将状态恢复为未完成
4. 系统恢复普通展示样式
5. 系统更新统计数量

#### 流程四：删除待办

1. 用户看到待办列表
2. 用户点击某条待办的删除按钮
3. 系统删除该待办
4. 系统更新列表和统计数量
5. 如果列表为空，系统展示空状态

#### 流程五：筛选待办

1. 用户看到全部待办列表
2. 用户点击“未完成”
3. 系统仅展示未完成事项
4. 用户点击“已完成”
5. 系统仅展示已完成事项
6. 用户点击“全部”
7. 系统展示所有事项

### 验收摘要

MVP 验收通过标准：

- 用户可以创建一条有效待办
- 空输入无法创建待办，并有明确提示
- 超长输入无法创建或被限制，并有明确提示
- 用户可以看到待办列表
- 用户可以完成和取消完成待办
- 用户可以删除待办
- 用户可以按全部、未完成、已完成筛选
- 页面能正确展示数量统计
- 无待办或筛选结果为空时有空状态提示
- 页面刷新后，已创建待办仍然存在
- 所有核心功能在桌面和移动端基本可用

作为普通用户，我希望看到所有已创建的待办事项，以便了解当前记录的任务。

#### US-005 标记完成

作为普通用户，我希望将待办标记为完成，以便区分已处理和未处理任务。

#### US-006 取消完成

作为普通用户，我希望把已完成待办恢复为未完成，以便修正误操作或重新跟进任务。

#### US-007 删除待办

作为普通用户，我希望删除不需要的待办，以便保持列表清爽。

#### US-008 按状态筛选

作为普通用户，我希望按全部、未完成、已完成筛选待办，以便快速查看特定状态任务。

作为普通用户，我希望看到待办数量统计，以便快速了解任务完成情况。

作为普通用户，我希望刷新页面后待办事项仍然存在，以便不用重复录入任务。

作为普通用户，我希望在没有待办或筛选结果为空时看到提示，以便理解当前列表为什么为空。

### 验收条件

#### US-001 新增待办

- Given 用户打开应用
- When 用户输入“买牛奶”并点击新增按钮
- Then 系统应创建一条标题为“买牛奶”的未完成待办
- And 列表中应���示该待办

#### US-002 空内容校验

- Given 用户打开应用
- When 用户未输入内容或只输入空格并点击新增按钮
- Then 系统不应创建待办
- And 应展示“请输入待办内容”之类的错误提示

#### US-003 长度限制

- Given 用户打开应用
- When 用户输入超过 100 个字符的标题并提交
- Then 系统应阻止提交或限制输入
- And 应展示“待办内容不能超过 100 个字符”之类的提示

#### US-004 查看待办列表

- Given 用户已创建多条待办
- When 用户打开应用首页
- Then 系统应展示所有待办事项
- And 每条待办应包含标题、完成状态和删除入口

#### US-005 标记完成

- Given 用户有一条未完成待办
- When 用户点击该待办的完成控件
- Then 系统应将该待办状态改为已完成
- And 该待办应展示完成样式

#### US-006 取消完成

- Given 用户有一条已完成待办
- When 用户点击该待办的完成控件
- Then 系统应将该待办状态改为未完成
- And 该待办应恢复未完成样式

#### US-007 删除待办

- Given 用户有一条待办
- When 用户点击删除按钮
- Then 系统应从列表中移除该待办
- And 统计数量应同步更新

#### US-008 按状态筛选

- Given 用户同时存在已完成和未完成待办
- When 用户点击“未完成”筛选
- Then 系统只展示未完成待办
- When 用户点击“已完成”筛选
- Then 系统只展示已完成待办
- When 用户点击“全部”筛选
- Then 系统展示全部待办

#### US-009 查看数量统计

- Given 用户有 3 条待办，其中 1 条已完成、2 条未完成
- When 用户查看页面统计区域
- Then 系统应展示全部 3、未完成 2、已完成 1

#### US-010 本地保存

- Given 用户已创建待办
- When 用户刷新浏览器页面
- Then 系统应从本地存储恢复待办列表
- And 已完成状态应保持一致

#### US-011 空状态提示

- Given 用户没有任何待办
- When 用户打开应用
- Then 系统应展示空状态提示
- Given 用户筛选“已完成”但没有已完成待办
- When 系统渲染列表
- Then 系统应展示当前筛选结果为空的提示

## 3. WWAS 条目

### WWAS-001 新增待办

- Why：用户需要快速记录待处理事项，避免遗忘。
- What：提供输入框、提交按钮和 Enter 快捷提交能力，创建新的未完成待办。
- Acceptance：
  - 输入有效文本后可以新增待办
  - 新增待办默认状态为未完成
  - 新增成功后输入框清空
  - 列表与统计数量立即更新
- Scope：
  - 单条文本待办创建
  - 标题去除首尾空格
  - 支持点击按钮提交
  - 支持 Enter 提交
- Out of Scope：
  - 富文本输入
  - 附件上传
  - 语音输入
  - AI 自动拆解任务

### WWAS-002 输入校验

- Why：防止无效数据进入待办列表，提升可用性和数据质量。
- What：对待办标题进行非空和长度校验。
- Acceptance：
  - 空字符串不能提交
  - 仅包含空格的字符串不能提交
  - 超过 100 字符时应阻止提交或给出提示
  - 校验失败时不新增待办
- Scope：
  - 前端输入校验
  - 明确错误提示
  - 提交前校验
- Out of Scope：
  - 服务端校验
  - 敏感词检测
  - 多语言内容审查

### WWAS-003 待办列表展示

- Why：用户需要看到已记录的任务，以便知道当前有哪些事项需要处理。
- What：展示待办事项列表，每条记录包含完成控件、标题和删除按钮。
- Acceptance：
  - 有数据时展示列表
  - 每条待办显示标题
  - 每条待办显示完成状态
  - 每条待办提供删除入口
- Scope：
  - 列表基础展示
  - 已完成和未完成样式区分
  - 按创建时间展示，默认新任务可显示在顶部或底部，需保持一致
- Out of Scope：
  - 拖拽排序
  - 分组展示
  - 分页加载
  - 虚拟列表

### WWAS-004 完成状态切换

- Why：用户需要区分任务是否已经完成。
- What：提供状态切换控件，支持未完成与已完成之间切换。
- Acceptance：
  - 点击未完成待办后变为已完成
  - 点击已完成待办后变为未完成
  - 完成状态变化后统计同步更新
  - 完成状态变化后写入本地存储
- Scope：
  - 单条待办状态切换
  - 完成样式展示
  - 状态持久化
- Out of Scope：
  - 批量完成
  - 完成时间记录
  - 完成历史记录

### WWAS-005 删除待办

- Why：用户需要移除错误、过期或不再需要的待办。
- What：提供单条待办删除功能。
- Acceptance：
  - 点击删除后，该待办从列表移除
  - 删除后统计数量更新
  - 删除后本地存储同步更新
  - 删除最后一条待办后展示空状态
- Scope：
  - 单条删除
  - 删除按钮
  - 删除后刷新视图
- Out of Scope：
  - 批量删除
  - 撤销删除
  - 回收站
  - 删除确认弹窗

### WWAS-006 筛选待办

- Why：当待办数量增多时，用户需要快速查看特定状态的任务。
- What：提供全部、未完成、已完成三个筛选入口。
- Acceptance：
  - 点击“全部”展示所有待办
  - 点击“未完成”仅展示未完成待办
  - 点击“已完成”仅展示已完成待办
  - 当前筛选状态有视觉高亮
- Scope：
  - 三种固定筛选状态
  - 客户端筛选
  - 筛选结果为空时展示提示
- Out of Scope：
  - 搜索
  - 标签筛选
  - 日期筛选
  - 自定义筛选条件

### WWAS-007 数量统计

- Why：用户需要快速了解任务总体进度。
- What：展示全部、未完成、已完成数量。
- Acceptance：
  - 新增待办后总数和未完成数增加
  - 完成待办后已完成数增加、未完成数减少
  - 删除待办后对应数量减少
  - 刷新页面后统计仍正确
- Scope：
  - 基础数量统计
  - 实时更新
- Out of Scope：
  - 图表统计
  - 周报/月报
  - 趋势分析

### WWAS-008 本地持久化

- Why：用户希望刷新页面后保留待办数据，避免重复录入。
- What：使用浏览器 localStorage 保存待办列表。
- Acceptance：
  - 新增、完成、取消完成、删除后均更新 localStorage
  - 页面刷新后恢复待办数据
  - 本地无数据时展示空状态
  - localStorage 数据异常时应用不崩溃
- Scope：
  - 浏览器本地存储
  - 单设备单浏览器数据保留
  - 异常数据容错
- Out of Scope：
  - 服务端数据库
  - 账号同步
  - 跨设备同步
  - 数据导入导出

### WWAS-009 空状态与错误提示

- Why：用户需要理解当前无数据或操作失败的原因。
- What：提供无待办、筛选结果为空和输入错误提示。
- Acceptance：
  - 无任何待办时展示空状态
  - 当前筛选无结果时展示筛选空状态
  - 输入为空时展示错误提示
  - 错误提示应清晰、可理解
- Scope：
  - 文案提示
  - 基础视觉状态
- Out of Scope：
  - 复杂插画
  - Toast 队列系统
  - 国际化文案

5. 验证其状态仍为已完成

#### P-002 不同浏览器数据隔离

- 验证浏览器 B 不显示浏览器 A 的 localStorage 数据

#### P-003 隐私模式访问

- 在隐私/无痕窗口打开应用
- 验证功能可用
- 关闭窗口后数据是否保留由浏览器策略决定，不作为产品保证

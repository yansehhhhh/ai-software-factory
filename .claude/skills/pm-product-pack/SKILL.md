# PM Product Pack Skill

## 目标

根据用户需求或已澄清需求，一次性生成可供 UI 设计、代码生成和测试使用的产品经理产物包。

## 自动化执行规则

- 本 skill 可在 `claude -p` 非交互模式下执行。
- 不要等待用户确认，不要反问，不要暂停流程。
- 如果信息不足，不要编造事实，把问题写入"待确认问题"。
- 只输出 Markdown。
- 输出应聚焦第一版可交付范围，避免过度设计。
- 内容必须能被后续 UI 设计、前后端代码生成和测试阶段消费。

## 输入

- 原始需求
- 需求讨论补充信息
- 目标输出目录（如调用方提供）

## 输出产物

在 AI 软件工厂主链路中，本 skill 的能力已整合进 `prd-generator`，由调用方按 `docs/${项目名称}/产品设计/` 拆分落盘，不再要求单独输出 `pm-product-pack.md` 或 `requirements.docx`。

独立使用本 skill 时，可以输出一个完整 Markdown 产物包；主链路使用时，应将 PRD、用户故事、WWAS、测试场景和样例数据分别归入 `PRD.md`、`UI-Design-Spec.md`、`04-流程图/` 和 `05-附录/术语表.md` 等拆分文件。

产物包内容结构：

```markdown
# PM 产品产物包

## 1. 产品需求文档 PRD

### 产品背景

### 目标用户

### 用户问题

### 产品目标

### 功能范围

### 非目标

### 核心用户流程

### 验收摘要

## 2. 用户故事 User Stories

### 用户角色

### Epic 列表

### 用户故事

使用格式：作为 <角色>，我希望 <能力>，以便 <价值>。

### 验收条件

优先使用 Given-When-Then。

### 优先级建议

## 3. WWAS 条目

每个条目使用：

- Why：为什么要做
- What：要交付什么
- Acceptance：如何验收
- Scope：范围
- Out of Scope：非范围

## 4. 测试场景 Test Scenarios

### 测试目标

### Happy Path

### 边界场景

### 异常与错误状态

### 权限与访问场景

### 数据校验场景

### 场景矩阵

## 5. 样例数据 Dummy Dataset

### 数据实体

### 字段定义

### 正常样例记录

### 边界样例记录

### 无效样例记录

### Demo 使用说明

### 测试使用说明

## 6. 待确认问题
```

**注意**：本 skill 不再生成 HTML 原型和原型说明。主链路中不要把全部内容塞进单个 `pm-product-pack.md`，而应由 `prd-generator` 拆分输出到产品设计目录；用户故事、WWAS、测试场景、样例数据等 PM 产物包内容优先进入 `05-附录/术语表.md`，必要摘要进入 PRD 和 UI 规范。

## 质量要求

- PRD 必须说明第一版 MVP 边界。
- 用户故事必须可以转化为开发任务。
- WWAS 必须包含明确验收标准。
- 测试场景必须覆盖正常、边界、异常和权限访问。 
- 样例数据必须覆盖正常、边界和无效数据。
- 如果需求涉及管理后台、表单、看板或工作流，必须体现页面/状态/数据流。
- 如果需求涉及 AI 能力，必须说明输入、输出、失败状态和人工复核点。
